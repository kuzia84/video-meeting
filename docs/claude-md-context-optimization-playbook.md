# Playbook: оптимизация CLAUDE.md и контекста Claude Code

Переносимая методика сокращения always-loaded контекста Claude Code без потери фактов.
Применима к любому репозиторию (моно- или обычному). Результат на этом проекте: три
CLAUDE.md сжаты с ~6950 до ~3740 слов (**~46%**), а бо́льшая часть детализации переведена
в файлы, которые грузятся только когда с ними работаешь.

---

## 0. Зачем

Корневой `CLAUDE.md` и `CLAUDE.md` уровня приложения **грузятся в контекст каждую сессию**.
Со временем они превращаются в архив дизайн-решений: длинные обоснования, дублирование между
файлами, пересказ того, что уже есть в `docs/` и в тестах. Это молча тратит контекст в каждом
запросе. Цель — чтобы CLAUDE.md **ориентировал** («что где лежит, что нельзя сломать, куда идти
за деталями»), а не документировал всё исчерпывающе.

Ключевая ментальная модель — разделить содержимое по «жанрам» и разложить по разным механизмам:

| Жанр контента                                                   | Куда                            | Когда грузится                                 |
| --------------------------------------------------------------- | ------------------------------- | ---------------------------------------------- |
| **Факты о коде** (модули, роуты, схема, как работает хранилище) | `CLAUDE.md` (иерархия)          | всегда (корень/app) или по локации (вложенные) |
| **Поведенческие директивы** («всегда делай X», «никогда Y»)     | `.claude/rules/*.md`            | always-on, либо lazy по `paths`                |
| **Пошаговые воркфлоу** (деплой, ревью, конкретная задача)       | `.claude/skills/`               | только при вызове                              |
| **Детерминированное принуждение** (проверки, напоминания)       | hooks в `.claude/settings.json` | на событие (Stop, PostToolUse, …)              |

Правило разделения: **факт живёт ровно в одном файле**. Если нужен обоим — в файле выше по
иерархии + ссылка, никогда не копия (копии расходятся).

## 1. Процесс (по шагам)

### Шаг 1. Инвентаризация

- Прочитай корневой `CLAUDE.md` и все `CLAUDE.md` уровня приложений.
- Замерь объём в словах — это и есть плата за контекст:
  ```bash
  for f in CLAUDE.md apps/*/CLAUDE.md; do echo "$f: $(wc -w < "$f") words"; done
  ```
- Помечай три класса лишнего:
  - **A. Reasoning**, которому место в комментарии кода или в `docs/`/тестах
    (напр. «этот `+1` load-bearing, потому что…» на 120 слов).
  - **B. Дублирование между файлами** (одно и то же объяснение в root и в app).
  - **C. Дублирование с `docs/`** (пересказ того, на что файл сам же ссылается).

### Шаг 2. Ввести правило иерархии

Добавь в корневой `CLAUDE.md` секцию «CLAUDE.md hierarchy» — она задаёт, кто чем владеет
(шаблон в §3). Это фундамент: все последующие переносы опираются на неё.

### Шаг 3. Сжать app-файлы и разнести детали по вложенным CLAUDE.md

- Раздели `src/` по естественным границам:
  - **бэкенд** — по модулям (`src/auth/`, `src/users/`, …);
  - **фронтенд** — по слоям (`src/app/` страницы, `src/components/`, `src/lib/` data-слой, `e2e/`),
    а не по каждому роуту.
- Для каждой границы создай вложенный `CLAUDE.md` с локальной детализацией.
- В app-файле оставь по модулю/слою **одну строку: что владеет + ссылка** на вложенный файл.
- Секцию «Commands» замени ссылкой на `package.json` (он — источник истины по скриптам);
  оставь только то, чего в `package.json` нет (аннотации «нужен Postgres», «своя БД», не-скриптовые
  команды типа запуска одного теста).

### Шаг 4. ⚠️ Верифицировать перед переносом (не пропускать)

**Нельзя переносить непроверенное** — доки дрейфуют. Перед тем как перекладывать текст, сверь
каждое утверждение с кодом. Эффективный способ — параллельные субагенты-исследователи, по одному
на модуль; они возвращают вердикт-лист, а не дампы файлов (дёшево по контексту):

```
Для каждого модуля запусти агента с задачей:
«Сверь эти утверждения из CLAUDE.md с кодом в <путь>. Для КАЖДОГО — вердикт
 CONFIRMED / WRONG / OUTDATED с file:line. Плюс: что есть в коде, но не в утверждениях.»
```

Типичные находки: устаревший статус фичи («фронта нет», а он есть), сместившаяся логика
(валидация переехала в общий компонент), неточные детали (имена функций, экспорты). Исправляй
их **при** переносе.

### Шаг 5. Вынести поведенческие директивы в `.claude/rules/`

`.claude/rules/*.md` — официальный авто-загружаемый механизм (docs → memory.md):

- **без** frontmatter `paths` → always-on (как корневой CLAUDE.md);
- **с** `paths: [glob…]` → **lazy: грузится только когда трогаешь подходящие файлы**;
- scope: project (`.claude/rules/`, в git) и user (`~/.claude/rules/`, для всех твоих проектов).

Важно про экономию: перенос always-on текста в always-on rule **токены не экономит** (стоит столько же).
Выигрыш дают только:

- **`paths`-scoping** — правило перестаёт висеть всегда и грузится, лишь когда релевантно
  (напр. «проверяй UI через Playwright MCP» → `paths: ["**/*.tsx", …]`);
- **user-level** — общая для всех проектов директива о том, _как тебе работать_
  (напр. «извлекай нужное, не читай файл целиком»), лежит один раз в `~/.claude/rules/`.

Оставляй в CLAUDE.md факты о коде; в rules переноси только «как действовать».

### Шаг 6. Закрепить процесс на будущее

Чтобы оптимизация не разъехалась обратно при росте проекта:

- **Соглашение**: в правиле иерархии — «новый модуль/слой **обязан** получить вложенный CLAUDE.md
  - ссылку в родителе» (may → must). Продублируй в правилах автономного цикла, если он есть.
- **Hook**: неблокирующее детерминированное напоминание (шаблон в §3.4) — сканирует директории
  модулей и предупреждает о тех, у кого нет `CLAUDE.md`.

## 2. Что оставлять, а что резать

Оставляй строку в CLAUDE.md, если она отвечает «**что мне нельзя сломать / куда идти**».
Убирай (в код-комментарий, в `docs/` или просто удаляй), если это «**почему так сделали**»
и рядом уже есть код/тест/спека/док, который это держит.

- ✅ Оставить: «`id`-tie-breaker load-bearing, e2e его пинит», «не ставь Content-Type на FormData»,
  ссылки на `docs/`, инварианты «row then bytes».
- ✂️ Убрать: замеры и разбор на 100+ слов, почему именно так; повторные объяснения; пересказ спеки,
  на которую сам же ссылаешься.

Резать можно агрессивно, если reasoning держится тестами/доками. Ничего не «теряется» — оно
остаётся там, где его и будут читать (у кода / в спеке).

## 3. Шаблоны (копировать)

### 3.1 Секция «CLAUDE.md hierarchy» (в корневой CLAUDE.md)

```markdown
## CLAUDE.md hierarchy

Each level owns a distinct scope:

- **Root** — monorepo-wide facts and the canonical version of anything cross-cutting.
- **`apps/*/CLAUDE.md`** — that app's own detail only; must not restate a root fact — link instead.
- Nested `CLAUDE.md` (`apps/api/src/<module>/`, `apps/web/src/{app,components,lib}`, …) own their
  detail, loaded into context only when you work there.

Rules that keep them from drifting:

- **A new backend module or top-level frontend layer MUST get its own nested `CLAUDE.md` in the same
  change, plus a one-line pointer in the parent** — don't grow the parent with the new module's detail.
- **A fact lives in exactly one file.** Cross-scope → the higher file + a reference; local → the local file.
- **When two files would both need it, put it higher and link — never copy.**
- Every nested file links **back** to its parent; each parent links **out** to its children.
```

### 3.2 Скелет вложенного CLAUDE.md

```markdown
# <Module/Layer> (`<path>`)

App-local detail for <what>. Parent: `<../CLAUDE.md>`.

## Responsibility

<Что владеет; единственный «источник истины» по чему-то, если так.>

## Routes / API / Public surface

<Список с кодами ответов / сигнатурами — коротко.>

## Load-bearing details

<Инварианты «не сломай»: одна строка на каждый, без длинных обоснований.
Ссылка на docs/тест, который это держит.>
```

### 3.3 Правило в `.claude/rules/` с `paths` (lazy)

```markdown
---
paths:
  - 'apps/web/src/**/*.tsx'
  - 'apps/web/src/app/**/*.css'
---

# <Behavioral rule title>

- <директива 1>
- <директива 2>
```

Без блока `---paths---` правило становится always-on — используй это только для действительно
всегда-применимых директив.

### 3.4 Неблокирующий Stop-хук покрытия документацией

`.claude/hooks/check-module-docs.sh`:

```bash
#!/usr/bin/env bash
# Non-blocking Stop hook: warn when a module/layer dir has no CLAUDE.md.
# Never blocks — always exits 0; surfaces a warning via JSON systemMessage.
set -uo pipefail
script_dir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
repo_root="$(cd "$script_dir/../.." && pwd)"; cd "$repo_root" || exit 0

missing=()
for base in apps/api/src apps/web/src; do          # ← подстрой под свой репозиторий
  [ -d "$base" ] || continue
  for dir in "$base"/*/; do
    [ -d "$dir" ] || continue
    [ -f "${dir}CLAUDE.md" ] || missing+=("${dir}CLAUDE.md")
  done
done
[ ${#missing[@]} -eq 0 ] && exit 0

msg="CLAUDE.md hierarchy: these dirs have no CLAUDE.md. Create one + a pointer in the parent:"
for m in "${missing[@]}"; do msg="${msg}"$'\n'"  - ${m}"; done
json_escape() { local s="$1"; s="${s//\\/\\\\}"; s="${s//\"/\\\"}"; s="${s//$'\n'/\\n}"; printf '%s' "$s"; }
printf '{"systemMessage": "%s"}\n' "$(json_escape "$msg")"
exit 0
```

Регистрация в `.claude/settings.json` (добавить группу, не трогая существующие хуки):

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/check-module-docs.sh",
            "statusMessage": "Checking module CLAUDE.md coverage..."
          }
        ]
      }
    ]
  }
}
```

Замечания: хуки читаются при старте сессии (активны со следующей). Хук **предупреждает**, а не
пишет доки сам — содержимое всё равно создаёт агент по усиленному правилу. `exit 0` = не блокирует
поток (в т.ч. автономные циклы).

## 4. Верификация результата

- Пересчитай слова (§1 шаг 1) — зафиксируй «было/стало».
- Проверь отсутствие висячих ссылок после сжатия:
  ```bash
  grep -rn "(below)\|(above)\|see below" CLAUDE.md apps/*/CLAUDE.md
  ```
- Проверь двустороннюю навигацию: каждый вложенный ссылается вверх, каждый родитель — вниз.
- Прогони lint/тесты (правки только в `.md`/конфигах ничего не ломают, но commit-хуки могут запускать
  проверки — убедись, что они зелёные).
- Валидируй JSON после правок `settings.json`:
  ```bash
  node -e 'JSON.parse(require("fs").readFileSync(".claude/settings.json","utf8"));console.log("ok")'
  ```

## 5. Подводные камни

- **Не переноси непроверенное.** Доки дрейфуют; всегда сверяй с кодом (Шаг 4).
- **Always-on rule ≠ экономия.** Он стоит столько же, сколько корневой CLAUDE.md. Экономят `paths`
  и user-level.
- **Не дроби фронтенд по роутам.** Один `src/app/CLAUDE.md` на страницы обычно лучше, чем файл на
  каждый роут; хук покрытия настраивай на **слои**, а не на каждую подпапку.
- **Не оставляй дубль.** Вынес в rule/вложенный файл — сократи источник до ссылки, иначе грузишь
  дважды.
- **YAML frontmatter в rules хрупок к форматтеру.** После автоформата проверь, что блок `paths`
  цел (иначе правило станет always-on).
- **User-level файлы не в git.** `~/.claude/rules/*` персональны и не поедут с репозиторием —
  командные правила клади в project-level `.claude/rules/`.
- **Коммит-сообщения:** не собирай их шелл-heredoc/here-string в неродном шелле — пиши в файл и
  `git commit -F <file>` (иначе маркеры делимитеров утекают в сообщение).
