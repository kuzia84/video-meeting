# Технический ресерч: загрузка файла встречи

**План:** [docs/meeting-file-upload-plan.md](./meeting-file-upload-plan.md)
**PRD:** [docs/meeting-file-upload.md](./meeting-file-upload.md)
**Дата:** 2026-07-16

Как реализовать план оптимально на текущем стеке. Решения с обоснованиями, подводные камни и открытые вопросы. Не замена плану: порядок работ там, здесь — «чем и почему».

## Что уже есть в проекте

Проверено по коду, а не по памяти:

| Факт                                                                                                   | Следствие для реализации                                             |
| ------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------- |
| `@nestjs/platform-express` 11 (Express, не Fastify)                                                    | Загрузка через multer — встроенный путь Nest                         |
| `@types/multer` **не установлен**                                                                      | Нужен: `npm i -D @types/multer -w @video-meetings/api`               |
| `MeetingsController` на CQRS (CommandBus/QueryBus), guard `JwtAuthGuard`                               | Файлы ложатся в ту же схему: команда на загрузку, запросы на чтение  |
| e2e на supertest, `maxWorkers: 1`, `testRegex: .e2e-spec.ts$`                                          | API-e2e добавляются рядом, без новой обвязки                         |
| `apps/web` — Next.js 15 + React 19, HeroUI, **страницы встречи нет**                                   | Фазы 4–5 требуют сперва создать саму страницу (см. «Пробел в плане») |
| `apps/web/src/lib/api/client.ts` — обёртка над `fetch`, жёстко ставит `Content-Type: application/json` | Для multipart её переиспользовать нельзя (см. «Прогресс»)            |
| JWT лежит в `sessionStorage`, уходит заголовком                                                        | Скачивание по `<a href>` не сработает: заголовка не будет            |

## Ключевые решения

| Вопрос                   | Решение                                | Почему                                                                                          |
| ------------------------ | -------------------------------------- | ----------------------------------------------------------------------------------------------- |
| Где буферизуется файл    | `diskStorage`, не память               | Дефолт multer — память: 100 МБ в RAM на каждую параллельную загрузку                            |
| Как режется лимит 100 МБ | `limits.fileSize` в multer             | Обрывает поток на лимите; `MaxFileSizeValidator` сработает уже после записи всех 101 МБ на диск |
| Имя файла на диске       | UUID, оригинал — в БД                  | `originalname` контролируется клиентом → path traversal                                         |
| Проверка владельца       | В **guard**, до интерцептора           | Иначе чужой запишет 100 МБ на диск и только потом получит 404                                   |
| Отдача файла             | `StreamableFile`                       | Не тянет файл в память, сохраняет работу интерцепторов                                          |
| Прогресс на фронте       | `XMLHttpRequest`                       | `fetch` не умеет прогресс отправки (см. ниже)                                                   |
| Скачивание на фронте     | `fetch` + blob + `URL.createObjectURL` | Токен уходит только заголовком                                                                  |

## Фаза 1. Хранение и загрузка

**Модель.** Отдельная `MeetingFile` со связью к `Meeting` и `onDelete: Cascade` — как `Meeting → User` сейчас. Поля: `id`, `meetingId`, `originalName`, `storedName` (UUID на диске), `size`, `mimeType`, `createdAt`. Каскад чистит только строки в БД; файлы на диске переживут удаление встречи — это уходит в задачу про управление встречами, здесь не решается.

**Каталог.** Через env (`UPLOAD_DIR`), с обновлением `apps/api/.env.example` — того требует CLAUDE.md. В Docker каталог обязан быть на примонтированном томе, иначе файлы умрут вместе с контейнером, а критерий фазы 2 «переживает перезапуск» — про перезапуск процесса, не пересоздание контейнера. Эти два случая стоит не путать.

**Конфигурация multer:**

```ts
MulterModule.register({
  storage: diskStorage({
    destination: process.env.UPLOAD_DIR,
    filename: (_req, _file, cb) => cb(null, randomUUID()),
  }),
  limits: { fileSize: 100 * 1024 * 1024, files: 1 },
});
```

`filename` с UUID — не косметика: `originalname` приходит от клиента и может содержать `../../etc/passwd`. Multer сам путь не санитизирует.

**Порядок guard → interceptor.** Nest выполняет guards до интерцепторов, поэтому `JwtAuthGuard` отсекает неавторизованного раньше, чем multer начнёт писать. А вот проверка владельца обычно живёт в обработчике — то есть уже **после** записи файла. Чужой пользователь успеет залить 100 МБ и получить 404 поверх записанного файла. Поэтому владельца надо проверять отдельным guard'ом (`MeetingOwnerGuard`) до интерцептора. Это же условие делает честным критерий «при отказе файл на диске не создаётся».

**Кириллица в имени файла.** Busboy отдаёт `originalname` в latin1, поэтому «Запись встречи.mp3» приедет как `Ð—Ð°Ð¿Ð¸ÑÑŒ...`. Лечится на входе:

```ts
const originalName = Buffer.from(file.originalname, 'latin1').toString('utf8');
```

Для русскоязычного продукта это не экзотика, а нормальный кейс — стоит закрыть тестом с кириллическим именем.

## Фаза 2. Список и скачивание

Отдача — `StreamableFile` (`docs.nestjs.com/techniques/streaming-files`):

```ts
return new StreamableFile(createReadStream(join(uploadDir, file.storedName)), {
  disposition: `attachment; filename*=UTF-8''${encodeURIComponent(file.originalName)}`,
});
```

`filename*=UTF-8''…` по RFC 5987, иначе кириллическое имя развалится уже в браузере. Голый `filename="…"` допускает только ASCII.

**Рассинхрон БД и диска.** Строка есть, файла нет (упал деплой, почистили каталог) — `createReadStream` бросит `ENOENT` уже внутри стрима, когда статус 200 отправлен. Клиент получит оборванную загрузку без внятной ошибки. Проверять существование до создания стрима и отдавать 404.

**Проверка принадлежности файла встрече** обязательна: `GET /meetings/:id/files/:fileId` не должен отдавать файл чужой встречи, даже если `:id` свой. Иначе перебором `fileId` вытекут чужие записи.

## Фаза 3. Валидация

**Размер.** Только `limits.fileSize`. Альтернатива из документации Nest — `ParseFilePipe` с `MaxFileSizeValidator` — здесь работает хуже: pipe запускается после того, как multer дописал файл, то есть 101 МБ уже лежат на диске, а время потрачено. `limits.fileSize` рвёт поток на 100 МБ.

**Текст ошибки.** Проверено по исходникам `@nestjs/platform-express/multer/multer/multer.utils.js`: `LIMIT_FILE_SIZE` превращается в `PayloadTooLargeException` с сообщением `File too large` — **лимит в тексте не назван**. PRD требует «сообщение, называющее лимит», значит нужен свой exception filter на `PayloadTooLargeException` либо перехват. Само собой это не появится.

**Тип.** `fileFilter` отклоняет до записи на диск — предпочтительнее `FileTypeValidator`, который сработает после. Но `fileFilter` судит по `mimetype`, а его присылает клиент, и подделать его тривиально. Настоящая проверка — по magic bytes; `FileTypeValidator` в Nest умеет их читать, но только из `file.buffer`, которого при `diskStorage` нет. То есть выбор такой:

- **Сейчас:** `fileFilter` по mimetype + расширению. Критерий PRD (`.exe` отклоняется) закрывается: браузер пришлёт для него не аудио-mimetype.
- **Честно:** дочитать первые байты записанного файла и сверить сигнатуру, затем удалить при несовпадении.

Первого достаточно: PRD явно выносит антивирус за скоуп, файлы не исполняются и доступны только владельцу. Но записать это как осознанный компромисс стоит — подделанный mimetype пройдёт.

**Файлы-сироты.** Multer пишет файл до обработчика. Любой отказ после записи — валидация, падение Prisma, отвалившаяся БД — оставит файл на диске навсегда. Критерий «при отказе не создаётся ни файл на диске, ни запись в базе» требует явной уборки: `unlink` в `catch`, и то же самое при обрыве соединения (`req.on('aborted')`). Само по себе это не работает никогда.

## Фазы 4–5. Фронтенд

**Прогресс отправки: `fetch` не подходит.** У `fetch` нет событий прогресса загрузки — только скачивания. Обходной путь через `ReadableStream` в теле с `duplex: 'half'` существует, но требует HTTP/2 и не поддержан Safari и Firefox. Рабочий вариант — `XMLHttpRequest`:

```ts
const xhr = new XMLHttpRequest();
xhr.upload.onprogress = (e) => e.lengthComputable && onProgress(e.loaded / e.total);
```

Отсюда следствие: `client.ts` для загрузки не переиспользуется — он построен на `fetch` и жёстко ставит `Content-Type: application/json`. Для `FormData` заголовок ставить **нельзя вообще**: браузер обязан подставить его сам вместе с `boundary`. Нужен отдельный аплоадер, а разбор ошибок (`ApiError`) стоит вынести из `client.ts` и переиспользовать в обоих.

**Скачивание.** Токен живёт в `sessionStorage` и уходит заголовком, поэтому `<a href="/meetings/…/files/…">` даст 401 — браузер заголовок не подставит. Значит `fetch` с заголовком → `blob()` → `URL.createObjectURL` → синтетический клик → `revokeObjectURL`. Для 100 МБ файл целиком окажется в памяти вкладки: терпимо, но не бесплатно. Альтернатива — короткоживущие подписанные ссылки, но это новый механизм, которого нет ни в PRD, ни в текущей аутентификации.

**Playwright.** `webServer` умеет поднимать несколько серверов массивом (web и API), `reuseExistingServer: !process.env.CI`. Загрузка — `setInputFiles` (в том числе из буфера, без файла на диске), скачивание — `page.waitForEvent('download')`, причём ожидание надо начинать **до** клика.

Риск для критерия про прогресс: на localhost 50 МБ уедут мгновенно, и прогресс может прыгнуть с 0 сразу в 100 — тест на «прогресс меняется» станет флейки. Лечится throttling через CDP или подменой ответа `page.route` с задержкой. Стоит заложить сразу, иначе тест будет мигать.

## Пробел в плане

**Страницы встречи в `apps/web` не существует.** Есть `login`, `register`, home и клиент `lib/api/meetings.ts` — но ни страницы списка встреч, ни страницы одной встречи. Фазы 4 и 5 сформулированы как «добавить на страницу встречи…», то есть предполагают то, чего нет.

Значит перед фазой 4 нужна работа, которой нет ни в PRD, ни в плане: маршрут `/meetings/[id]`, загрузка встречи, состояния загрузки и ошибки, защита от неавторизованного доступа. Это отдельная задача — трогать её в рамках этой фичи означало бы протащить в неё пол-фронтенда. Решение за владельцем продукта: либо она делается до фазы 4 отдельно, либо PRD расширяется явно.

## Открытые вопросы

- **Проверка `Content-Length` до приёма тела.** Можно отклонять заведомо большие запросы по заголовку, не читая поток. Заголовок подделывается, поэтому `limits.fileSize` всё равно нужен — но это дешёвый ранний отказ. Нужно ли — решать по нагрузке.
- **Параллельные загрузки.** PRD молчит о том, сколько их может идти разом. Пять по 100 МБ — это 500 МБ записи на диск конкуренцией; при «ограниченных ресурсах» из PRD может понадобиться лимит на пользователя.
- **Генерация файла 101 МБ в тестах.** `Buffer.alloc(101 * 1024 * 1024)` съест 101 МБ RAM у jest. Дешевле разрежённый файл через `fs.truncate` — создаётся мгновенно и почти не занимает места.
