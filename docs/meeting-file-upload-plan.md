# План: Загрузка файла встречи

**PRD:** [docs/meeting-file-upload.md](./meeting-file-upload.md)
**Дата:** 2026-07-16

Каждая фаза закрывается собственными e2e-тестами: они пишутся вместе с кодом фазы и проверяют её критерий готовности. Отдельной фазы «написать тесты» нет.

Зависимости от фичи управления встречами ([docs/meeting-management-plan.md](./meeting-management-plan.md)): фазы 1–3 самодостаточны, а фазы 4–5 требуют её фазы 1 (обвязка браузерных e2e поднимается там) и фазы 2 (страница встречи, на которую ложится блок файлов). Согласованный порядок работ: фазы 1–3 отсюда → управление встречами → фазы 4–5 отсюда.

## Эндпоинты API

Ориентир для реализации, а не жёсткий контракт: детали уточняются при разработке. Все эндпоинты защищены JWT и работают только для владельца встречи — чужая встреча отвечает так же, как несуществующая.

| Метод и путь                      | Назначение                        | Ответ                                                       |
| --------------------------------- | --------------------------------- | ----------------------------------------------------------- |
| `POST /meetings/:id/files`        | Загрузка одного файла (multipart) | Метаданные созданного файла: id, имя, размер, дата загрузки |
| `GET /meetings/:id/files`         | Список файлов встречи             | Массив метаданных; пустой массив, если файлов нет           |
| `GET /meetings/:id/files/:fileId` | Скачивание файла                  | Поток содержимого файла                                     |

Отказы: неподдерживаемый тип или превышение лимита размера — ошибка валидации с причиной в теле; отсутствие или чужая встреча/файл — не найдено; запрос без валидного токена — не авторизован.

Удаление файлов (`DELETE`) приходит из фичи управления встречами.

## Фаза 1. Хранение файла и загрузка через API

**Цель:** Владелец встречи может загрузить файл: содержимое ложится на диск сервера, метаданные — в базу.
**Затрагивает:** database, backend

**Задачи:**

- Добавить модель файла встречи (имя, размер, дата загрузки, путь на диске) со связью к `Meeting` и миграцию.
- Настроить каталог хранения файлов на диске рядом с API, с изолированным каталогом для тестового прогона.
- Реализовать `POST /meetings/:id/files` с приёмом одного multipart-файла: запись содержимого на диск, метаданных — в базу.
- Закрыть эндпоинт JWT и проверкой владельца встречи: чужая и несуществующая встреча отвечают одинаково — не найдено.
- Написать API-e2e: успешная загрузка владельцем, отказ чужому пользователю, отказ без токена.

**Критерий готовности:** API-e2e фазы зелёные на чистой базе: загрузка с токеном владельца возвращает метаданные и кладёт содержимое на диск, а запросы чужого пользователя и без токена файл не создают.

## Фаза 2. Список и скачивание файлов через API

**Цель:** Ранее загруженные файлы можно перечислить и получить их содержимое обратно без потерь.
**Затрагивает:** backend

**Задачи:**

- Реализовать `GET /meetings/:id/files`: массив метаданных файлов встречи, пустой массив при их отсутствии.
- Реализовать `GET /meetings/:id/files/:fileId`: отдача содержимого потоком с исходным именем файла.
- Применить к обоим эндпоинтам JWT и проверку владельца, включая принадлежность файла этой встрече.
- Написать API-e2e: список после загрузки, побайтовая сверка скачанного с исходным, сохранность файла после перезапуска приложения, отказы чужому и без токена.

**Критерий готовности:** API-e2e фазы зелёные: скачанное содержимое побайтово совпадает с исходным и файл остаётся в списке после перезапуска приложения, а чужому пользователю и без токена оба эндпоинта содержимого не отдают.

## Фаза 3. Валидация типа и размера при загрузке

**Цель:** Неподдерживаемые и слишком большие файлы отклоняются с внятной причиной и не попадают в хранилище.
**Затрагивает:** backend

**Задачи:**

- Ограничить размер загружаемого файла 100 МБ, отклоняя превышение ошибкой валидации с указанием лимита в теле ответа.
- Ограничить типы принимаемых файлов до mp3, wav, m4a, mp4, отклоняя остальные ошибкой валидации с указанием причины.
- Гарантировать, что при отказе не создаётся ни файл на диске, ни запись в базе, а ранее загруженные файлы не затрагиваются.
- Написать API-e2e: отказ на 101 МБ, отказ на .exe, неизменность списка файлов после обеих попыток.

**Критерий готовности:** API-e2e фазы зелёные: загрузка 101 МБ и загрузка .exe отклонены с сообщением, называющим соответственно лимит и тип, и список файлов встречи после обеих попыток не изменился.

## Фаза 4. Список файлов на странице встречи

**Цель:** Владелец видит на странице встречи все свои загруженные файлы и может скачать любой из них.
**Затрагивает:** frontend

**Задачи:**

- Добавить на страницу встречи блок списка файлов с именем, размером и датой загрузки.
- Добавить действие скачивания файла из списка.
- Показывать пустое состояние, когда файлов у встречи нет.
- Написать браузерные e2e: список с корректными метаданными, скачивание по клику, пустое состояние.

**Критерий готовности:** Браузерные e2e фазы зелёные: на встрече с файлами виден список с корректными именем, размером и датой и файл скачивается по клику, а на встрече без файлов показано пустое состояние, а не ошибка.

## Фаза 5. Загрузка файла из интерфейса с прогрессом

**Цель:** Владелец загружает файл прямо со страницы встречи и видит ход загрузки и причину отказа.
**Затрагивает:** frontend

**Задачи:**

- Добавить на страницу встречи выбор одного файла и его отправку в эндпоинт загрузки.
- Показывать индикатор прогресса, отражающий реальный ход отправки.
- Обновлять список файлов встречи по завершении загрузки.
- Показывать сообщение об отказе по типу и по размеру с причиной, полученной от API.
- Написать браузерные e2e: прогресс при загрузке крупного файла, появление файла в списке по завершении, показ причины отказа по типу и по размеру.

**Критерий готовности:** Браузерные e2e фазы зелёные: при загрузке файла со страницы встречи прогресс меняется, по завершении файл появляется в списке, а при отказе по типу или размеру на странице виден текст причины.
