# План: Управление встречами

**PRD:** [docs/meeting-management.md](./meeting-management.md)
**Дата:** 2026-07-16

Порядок фаз учитывает зависимость от фичи загрузки: фазы 1–7 самодостаточны, фаза 8 требует готового API загрузки (фазы 1–3 плана [docs/meeting-file-upload-plan.md](./meeting-file-upload-plan.md)).

## Эндпоинты API

Ориентир для реализации, а не жёсткий контракт: детали уточняются при разработке. Все ручки под JWT и работают только для владельца встречи — чужая встреча отвечает так же, как несуществующая.

| Метод и путь                         | Назначение                           | Ответ               |
| ------------------------------------ | ------------------------------------ | ------------------- |
| `PATCH /meetings/:id`                | Правка названия, описания, времени   | Обновлённая встреча |
| `DELETE /meetings/:id`               | Удаление встречи со всеми её файлами | Пустой ответ        |
| `DELETE /meetings/:id/files/:fileId` | Удаление одного файла встречи        | Пустой ответ        |

Существующие `POST /meetings`, `GET /meetings`, `GET /meetings/:id` не меняются. Ручки файлов (`POST`/`GET`) приходят из фичи загрузки.

Отказы: ошибка валидации при пустом названии или окончании раньше начала; отсутствующая либо чужая встреча или файл — не найдено; запрос без валидного токена — не авторизован.

## Фаза 1. Список встреч на главной

**Цель:** Владелец видит на главной все свои встречи и может уйти к созданию новой.
**Затрагивает:** frontend, инфраструктура

**Задачи:**

- Поднять обвязку автоматических браузерных e2e в `apps/web` (`@playwright/test`) с прогоном против поднятых web и API. Это не Playwright MCP из `docs/architecture/frontend-ui.md`: тот водит браузер вручную для проверки вёрстки, здесь нужны коммитимые тесты, которые гоняются сами.
- Заменить блок трёх последних встреч на главной полным списком с постраничной навигацией.
- Показывать пустое состояние с приглашением создать первую встречу, когда встреч нет.
- Добавить на главную кнопку перехода к созданию встречи.
- Написать браузерные e2e: список с пагинацией, пустое состояние, переход по кнопке.

**Критерий готовности:** Браузерные e2e фазы зелёные: у пользователя с 25 встречами видна первая страница списка и переход на следующую показывает следующую порцию, у пользователя без встреч — пустое состояние, а кнопка ведёт на страницу создания.

## Фаза 2. Страница встречи

**Цель:** Владелец открывает встречу из списка и видит её данные.
**Затрагивает:** frontend

**Задачи:**

- Добавить маршрут страницы отдельной встречи и загрузку встречи по идентификатору.
- Показывать название, описание, время начала и окончания.
- Показывать состояния загрузки, сетевой ошибки и «встреча не найдена».
- Отправлять неавторизованного пользователя на страницу входа.
- Написать браузерные e2e: открытие своей встречи из списка, чужая встреча как ненайденная, редирект неавторизованного.

**Критерий готовности:** Браузерные e2e фазы зелёные: владелец открывает встречу из списка и видит её данные, чужая встреча показывается как ненайденная, а неавторизованный попадает на страницу входа.

## Фаза 3. Форма создания встречи

**Цель:** Владелец заводит встречу через интерфейс, а не запросом к API.
**Затрагивает:** frontend

**Задачи:**

- Добавить отдельную страницу создания встречи с полями: название, описание, время начала, время окончания.
- Проверять обязательность названия и что окончание позже начала, показывая причину у поля.
- Создавать встречу по отправке формы и переводить пользователя на её страницу.
- Отправлять неавторизованного пользователя на страницу входа.
- Написать браузерные e2e: успешное создание, отказ на пустом названии, отказ на окончании раньше начала.

**Критерий готовности:** Браузерные e2e фазы зелёные: корректная форма создаёт встречу и приводит на её страницу с введёнными данными, а пустое название и окончание раньше начала показывают причину у поля и встречу не создают.

## Фаза 4. Редактирование встречи в API

**Цель:** Данные встречи можно изменить запросом, с теми же правилами, что и при создании.
**Затрагивает:** backend

**Задачи:**

- Реализовать правку названия, описания и времени встречи как команду CQRS.
- Применить к ней те же правила валидации, что и при создании, включая «окончание позже начала».
- Закрыть операцию проверкой владельца: чужая и несуществующая встреча отвечают одинаково.
- Написать API-e2e: успешная правка владельцем, отказ по валидации, отказ чужому пользователю, отказ без токена.

**Критерий готовности:** API-e2e фазы зелёные: владелец меняет данные встречи и получает их обновлёнными, некорректное время отклоняется, а чужой пользователь и запрос без токена встречу не меняют.

## Фаза 5. Редактирование встречи в интерфейсе

**Цель:** Владелец правит встречу со страницы встречи и видит результат.
**Затрагивает:** frontend

**Задачи:**

- Добавить на страницу встречи форму правки названия, описания и времени с текущими значениями.
- Показывать причину отказа у поля при некорректных данных.
- Обновлять данные встречи на странице после сохранения.
- Написать браузерные e2e: правка сохраняется и видна после перезагрузки, некорректное время показывает причину.

**Критерий готовности:** Браузерные e2e фазы зелёные: изменённые название, описание и время видны после перезагрузки страницы встречи, а некорректное время показывает причину у поля и не сохраняется.

## Фаза 6. Удаление встречи и файла в API с уборкой диска

**Цель:** Удаление убирает и записи, и файлы с диска, не оставляя занятого места.
**Затрагивает:** backend, database

**Задачи:**

- Реализовать удаление одного файла встречи: запись и файл с диска.
- Реализовать удаление встречи: её записи, записи её файлов и все её файлы с диска.
- Вынести уборку диска в один механизм, общий для обоих случаев.
- Выполнять удаление в порядке, оставляющем при сбое осиротевший файл, а не запись без файла.
- Написать API-e2e: удаление файла, удаление встречи с несколькими файлами, отказы чужому и без токена, отсутствие файлов на диске после удаления.

**Критерий готовности:** API-e2e фазы зелёные: после удаления файла и после удаления встречи с двумя файлами каталог хранения не содержит ни одного их файла, а чужой пользователь и запрос без токена ничего не удаляют.

## Фаза 7. Удаление в интерфейсе с подтверждениями

**Цель:** Владелец удаляет файл или встречу из интерфейса и не может сделать это случайно.
**Затрагивает:** frontend

**Задачи:**

- Добавить удаление отдельного файла на странице встречи с диалогом подтверждения.
- Добавить удаление встречи с диалогом подтверждения и переходом на главную после удаления.
- Оставлять данные нетронутыми при отмене в диалоге.
- Написать браузерные e2e: удаление файла, удаление встречи, отмена в обоих диалогах.

**Критерий готовности:** Браузерные e2e фазы зелёные: подтверждённое удаление убирает файл из списка и встречу с главной, а отмена в обоих диалогах не удаляет ничего.

## Фаза 8. Шаг загрузки файлов в форме создания

**Цель:** Выбранные в форме файлы попадают во встречу сразу после её создания.
**Затрагивает:** frontend

**Задачи:**

- Добавить в форму создания выбор файлов встречи.
- После создания встречи загружать выбранные файлы с показом прогресса.
- Переводить пользователя на страницу встречи по завершении загрузки.
- Показывать сообщение о неудавшейся загрузке, оставляя созданную встречу доступной.
- Написать браузерные e2e: создание с файлом и прогрессом, падение загрузки при уже созданной встрече.

**Критерий готовности:** Браузерные e2e фазы зелёные: форма с файлом создаёт встречу, показывает прогресс и приводит на страницу встречи с файлом в списке, а при неудавшейся загрузке встреча открыта и видно сообщение об ошибке.
