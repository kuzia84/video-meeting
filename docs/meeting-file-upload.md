# Загрузка файла встречи

**Дата:** 2026-07-16
**Статус:** draft

## Цель

Пользователю нужно приложить к своей встрече запись (аудио или видео), чтобы она сохранилась и позже была отправлена на транскрипцию и ИИ-анализ. Без этого шага у продукта нет входной точки: анализировать нечего.

## Пользовательские сценарии

- **Пользователь:** создатель встречи — **Действие:** открывает страницу своей встречи и выбирает файл записи для загрузки — **Результат:** файл загружается, по завершении появляется в списке файлов этой встречи с именем, размером и датой загрузки.
- **Пользователь:** создатель встречи — **Действие:** во время загрузки крупного файла смотрит на страницу — **Результат:** видит индикатор прогресса и понимает, что загрузка идёт, а не зависла.
- **Пользователь:** создатель встречи — **Действие:** выбирает файл неподдерживаемого типа или больше 100 МБ — **Результат:** получает понятное сообщение об отказе с указанием причины, файл не сохраняется, ранее загруженные файлы не затрагиваются.
- **Пользователь:** создатель встречи — **Действие:** возвращается на страницу встречи позже — **Результат:** видит все ранее загруженные файлы этой встречи и может скачать любой из них.
- **Пользователь:** авторизованный пользователь, не создатель встречи — **Действие:** пытается открыть чужую встречу или её файл — **Результат:** доступа не получает.

## Скоуп

**В скоупе:**

- Загрузка одного файла за раз в конкретную встречу её создателем.
- Приём аудио- и видеофайлов (mp3, wav, m4a, mp4) размером до 100 МБ.
- Отказ с внятным сообщением при неподдерживаемом типе и при превышении лимита размера.
- Индикатор прогресса загрузки в интерфейсе.
- Список загруженных файлов на странице встречи: имя, размер, дата загрузки.
- Скачивание ранее загруженного файла создателем встречи.
- Файлы встречи видны и доступны только её создателю.
- E2E-тесты на весь сценарий загрузки: успешная загрузка, отказы по типу и размеру, скачивание, разграничение доступа.
- Браузерные e2e-тесты на интерфейс загрузки: прогресс, появление файла в списке, пустое состояние, показ причины отказа.

**Не в скоупе:**

- Сама страница встречи и обвязка браузерных тестов — создаются в фиче управления встречами ([docs/meeting-management.md](./meeting-management.md)), здесь предполагаются готовыми.
- Транскрипция, ИИ-анализ и любая обработка содержимого файла — это следующая итерация; здесь только хранение.
- Переименование загруженных файлов.
- Удаление файлов и удаление встречи вместе с ними — описано в [docs/meeting-management.md](./meeting-management.md).
- Множественная загрузка за раз и drag-and-drop.
- Докачка при обрыве соединения и возобновляемая загрузка.
- Воспроизведение и превью файла в браузере.
- Собственное или облачное хранилище (S3 и аналоги) — используем диск сервера.
- Версии файла и история изменений.
- Доступ участников встречи к файлам и шаринг наружу — модели участников сейчас нет.
- Квоты на суммарный объём файлов пользователя.
- Антивирусная проверка загруженного содержимого.

## Технические ограничения

- Файлы хранятся на диске сервера рядом с API — объём ограничен диском конкретной машины и не переживает пересоздание контейнера без примонтированного тома.
- У встречи сейчас есть только владелец (`Meeting` связан с `User` один-ко-многим); сущности участников не существует, поэтому любая модель доступа шире «только владелец» требует отдельной работы.
- Ограничение в 100 МБ означает, что часовая запись встречи в исходном качестве может не поместиться — пользователю придётся сжимать файл самому.

## Тестирование

E2E-тесты обязательны и входят в определение готовности — фича не принимается без них. Тесты пишутся вместе с кодом той фазы, чей критерий они проверяют, а не откладываются на конец.

Два уровня:

- **API-уровень** — реальные HTTP-запросы к поднятому приложению: загрузка, список, скачивание, негативные случаи (лимит, тип) и разграничение между двумя пользователями.
- **Браузерный уровень** — прогон интерфейса в реальном браузере: прогресс загрузки, появление файла в списке, пустое состояние, текст причины отказа. Обвязка для браузерных тестов поднимается в фиче управления встречами ([docs/meeting-management.md](./meeting-management.md)), которая делается раньше, — здесь она уже есть.

Юнит-тесты пишутся по необходимости и обязательными не считаются.

## Критерии готовности

- [ ] Создатель встречи загружает mp3 размером ~10 МБ и после завершения видит его в списке файлов этой встречи с корректными именем, размером и датой.
- [ ] Загруженный файл переживает перезапуск API: после рестарта он по-прежнему в списке и скачивается.
- [ ] Скачанный файл побайтово совпадает с исходным.
- [ ] Попытка загрузить файл размером 101 МБ отклоняется с сообщением, называющим лимит; файл в списке не появляется.
- [ ] Попытка загрузить .exe (или другой неподдерживаемый тип) отклоняется с сообщением о типе; файл в списке не появляется.
- [ ] Во время загрузки файла ~50 МБ в интерфейсе виден прогресс, который меняется, а по завершении сменяется файлом в списке.
- [ ] Пользователь Б, открывая встречу пользователя А по прямой ссылке, не видит её файлов и не может скачать файл по прямой ссылке на него.
- [ ] Неавторизованный запрос на загрузку и на скачивание файла отклоняется.
- [ ] Страница встречи без загруженных файлов показывает пустое состояние, а не ошибку.
- [ ] API-e2e покрывают каждый из трёх эндпоинтов, оба негативных случая (лимит, тип) и попытку доступа чужого пользователя.
- [ ] Браузерные e2e покрывают загрузку с прогрессом, появление файла в списке, пустое состояние и показ причины отказа.
- [ ] `npm run test -w @video-meetings/api`, API-e2e и браузерные e2e проходят зелёными на чистой базе.
