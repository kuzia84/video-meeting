# Ralph-цикл: последовательное выполнение нескольких milestone

**Дата:** 2026-07-19
**Статус:** approved

## Цель

Сейчас Ralph-цикл (`.claude/hooks/stop.js`) проходит **один** milestone: пока в нём есть открытые issue — подаёт их Claude по одной, когда issue кончаются — создаёт PR и останавливается. Нужно, чтобы за один прогон цикл проходил **несколько milestone подряд** (например, Фаза 3 → Фаза 4 → Фаза 5), а PR создавался один — на всю цепочку.

## Решения (из брейншторма)

- **Одна ветка, один PR на всю цепочку.** Все milestone цепочки делаются на одной ветке; когда закрыты все issue всех milestone — один PR со всеми issue. Зависимости между фазами решаются сами (всё в одной ветке). Прецедент в репозитории: `feat/meeting-management-phases-5-8`.
- **Явный упорядоченный список milestone в конфиге.** Цикл идёт строго в заданном порядке.

## Формат конфига

```json
{
  "milestones": [
    "Профиль пользователя — Фаза 3. Редактирование имени",
    "Профиль пользователя — Фаза 4. Загрузка аватара: хранение и отдача"
  ],
  "branch": "feature/user-profile-phases-3-4",
  "prTitle": "Профиль пользователя — Фазы 3–4",
  "maxIterations": 10,
  "maxSessionBytes": 1500000,
  "prompt": "..."
}
```

- `milestones` — упорядоченный список; цикл идёт в этом порядке.
- **Обратная совместимость:** старое поле `milestone` (строка) работает как цепочка из одного элемента. Если задано `milestones` — оно главнее. Пусты оба → цикл выключен.
- `prTitle` — **необязательный** заголовок PR. Если не задан: при одном milestone — его имя (текущее поведение), при нескольких — имена через « + ».
- Остальные поля (`branch`, `maxIterations`, `maxSessionBytes`, `prompt`) — без изменений.

## Логика прохода

Каждое срабатывание хука:

1. Для **каждого** milestone цепочки запрашивает открытые issue (`gh issue list`, N вызовов на N milestone).
2. Считает:
   - **активный milestone** = первый в списке, где ещё есть открытые issue;
   - **следующая issue** = issue активного milestone с наименьшим номером (порядок плана, как сейчас);
   - **суммарно открытых** = сумма открытых issue по всем milestone цепочки.
3. Если суммарно открытых > 0:
   - через `decide()` (существующая чистая функция, без изменений) с `openCount = суммарно открытых` получает вердикт `continue` / `stall` / `limit` / `session`;
   - на `continue` — блокирует остановку, подаёт следующую issue; в `{milestone}` промпта подставляется **активный** milestone.
4. Если суммарно открытых == 0 → создаёт **один** PR на всю цепочку и останавливается.

## Почему прогресс считается по сумме, а не по активному milestone

Сталл-гард ловит буксование так: если мы в цепочке хук-продолжений и число открытых issue **не убавилось** — Claude застрял. Если мерить число открытых по **активному** milestone, то на стыке фаз (Фаза 3 опустела → активной стала Фаза 4) счётчик прыгнет вверх (0 → 5) и сталл-гард **ложно** сработает. **Сумма** по всей цепочке убывает монотонно через все переходы, поэтому прогресс меряется ею. «Какую issue подать» (активный milestone) и «есть ли прогресс» (сумма) — две независимые величины.

## PR на всю цепочку

Когда суммарно открытых == 0, создаётся один PR (те же предохранители, что и сейчас: HEAD на нужной ветке, есть коммиты поверх базовой, PR ещё не открыт):

- **Заголовок:** `prTitle`, либо (по умолчанию) имя единственного milestone / имена через « + ».
- **Тело:** секция `## <имя milestone>` на каждый milestone цепочки со списком его **закрытых** issue, затем строка «🤖 Generated with Claude Code».

## Границы (что НЕ меняется)

- `decide()` — чистая функция стал/лимит/сессия — без изменений, переиспользуется как есть.
- `maxIterations` считает задачи по всей цепочке за прогон (не по milestone). `maxSessionBytes` — без изменений.
- HITL (`maxIterations: 1`) по-прежнему пауза после каждой задачи, теперь и через границы milestone.
- Формат файла состояния (`.ralph-state.json`), правила `ralph.md`, обвязка в `settings.json` — без изменений.

## Тестируемость

Новая чистая функция `selectActive(перечень [{ milestone, issues[] }])` → `{ activeMilestone, nextIssue, totalOpen }`, покрытая юнит-тестами:

- несколько milestone, у первого есть открытые → активен первый;
- первый пуст, второй непустой → активен второй (переход);
- все пусты → `totalOpen == 0`, `activeMilestone == null`;
- порядок issue внутри активного milestone — по возрастанию номера.

`decide()` уже покрыт; переиспользуется без изменений.

## Критерии готовности

- [ ] `milestones: [A, B]` в конфиге: цикл проходит A до конца, затем B, на одной ветке.
- [ ] На стыке A→B сталл-гард не срабатывает ложно (проверяется юнит-тестом `selectActive` + логикой суммы).
- [ ] Когда закрыты все issue A и B — создаётся ровно один PR, в теле секции по A и по B со списком закрытых issue.
- [ ] Старый конфиг с полем `milestone` (строка) продолжает работать без изменений.
- [ ] `maxIterations`/`maxSessionBytes`/HITL работают через границы milestone.
