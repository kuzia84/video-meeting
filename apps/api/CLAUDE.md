# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@video-meetings/api` — NestJS 11 backend, part of the `video-meetings` monorepo (see root `CLAUDE.md` for cross-workspace commands/architecture). Port 3001 (`process.env.PORT` overrides).

## Commands

Scripts are in this workspace's `package.json`; run from here or from the root with `-w @video-meetings/api`. Notes beyond the script names: `npm run test:e2e` needs Postgres (`npm run db:up`) and uses its **own** database (see below); `npm run db:seed` writes the fixed dev account (below); `npm run db:clean-e2e` reclaims test litter (below). Not scripts: `npx prisma migrate dev` (apply/create migrations against the running Postgres); a single test via `npx jest app.controller.spec.ts` or `npx jest -t 'name'`.

**Fixed dev account** (`prisma/seed.ts`): `test@mail.com` / `Pasword1!`, plus three sample meetings. Idempotent (upserts the user, replaces its own three meetings by title). Survives `npm run test:e2e` (separate database). Nothing in tests depends on it — it's only for poking at the app by hand. The password guards only a local dev database; never reuse it anywhere real.

## Architecture

- **Bootstrap** (`src/main.ts`, `src/app.module.ts`): CORS allowlist from `CORS_ORIGINS` (comma-separated, default `http://localhost:3000`), `app.enableShutdownHooks()` (Prisma disconnect on SIGTERM/SIGINT), global `ValidationPipe({ whitelist: true, transform: true })`. `ConfigModule.forRoot({ isGlobal: true })` reads `apps/api/.env` (see `.env.example`: `DATABASE_URL`, `JWT_SECRET`, `JWT_EXPIRES_IN` default `1h`, `CORS_ORIGINS`, `UPLOAD_DIR` default `./uploads`).
- **Shared package**: resolved from compiled `dist/` (jest is the exception — resolves source via `moduleNameMapper`). Full rationale in root `CLAUDE.md` "Shared package"; the API imports both types (`ApiResponse<T>`, `UserProfile`, …) and runtime values (`pickAvatarColorName`, …).
- **TS config**: `tsconfig.json` extends root base + NestJS options (`emitDecoratorMetadata`, `experimentalDecorators`, `nodenext`); `tsconfig.build.json` excludes `test/`/`*.spec.ts`. Jest config inline in `package.json` (unit); e2e uses `test/jest-e2e.json`.
- **CQRS**: feature modules use `@nestjs/cqrs`. **Read `docs/architecture/cqrs.md` before adding an operation** — Command-vs-Query rule, naming, error propagation, and the recipe for a new command/query.

### Modules

- **Auth (`src/auth/`)** — register/login/change-password, CQRS. Routes: `POST /auth/register` (201), `POST /auth/login` (200), `POST /auth/change-password` (204, JWT-guarded). Issues JWT via `TokenService`; hashes with `bcrypt` at shared `BCRYPT_ROUNDS` (10). Passwords validated by the shared `IsPassword()` decorator (8–72 chars) so register and change-password can't drift. **Auth never touches the DB directly** — dispatches `CreateUserCommand`/`GetUserByEmailQuery`/`GetUserByIdQuery`/`UpdateUserPasswordCommand` to Users over the bus; only hashing + the login timing-attack dummy hash stay here. Also **provides + exports** `JwtAuthGuard` (`src/auth/guards/`), the `@CurrentUser()` decorator, and the `AuthUser` type. Error language: 401 `Invalid credentials` (same for unknown email / wrong password), 409 duplicate email. Details: `docs/architecture/cqrs.md` + specs `2026-07-14-auth-cqrs-refactor-design.md`, `2026-07-15-auth-users-split-design.md`.
- **Users (`src/users/`)** — **the only place `prisma.user` is touched.** CQRS handlers: `CreateUserCommand` (assigns `avatarColor` via `pickAvatarColorName()`; `P2002` → 409 `email`), `UpdateUserNameCommand`, `UpdateUserPasswordCommand` (takes a ready hash — Users is unaware of bcrypt), `GetUserByEmailQuery`, `GetUserByIdQuery` (both return the full `User` **incl. `passwordHash`** as an internal contract — never serialize directly; narrow to `UserProfile` via `to-profile.ts`). Profile routes (JWT-guarded): `GET /users/me` (200), `PATCH /users/me` (200, sets display name, `UpdateProfileDto` 1..100 trimmed). A valid token whose account was deleted → **401** (session invalid), not 404. **Avatar (`src/users/avatar/`)**: `POST /users/me/avatar` (JPEG/PNG/WebP, 5 MB, validated by extension+mimetype filter **and** a magic-byte content sniff), `GET /users/me/avatar` (streams bytes, auth-only — the frontend fetches as a blob with the auth header). Stored under the **`avatars/` subdirectory** of `UPLOAD_DIR` by `AvatarStorage`, UUID names (subdir so the meeting-file orphan sweep never confuses the two). Concurrent replaces can strand one file — an accepted ≤5 MB leak reclaimed by the sweep (below).
- **Meetings (`src/meetings/`)** — per-user meetings, CQRS, JWT-guarded via the exported `JwtAuthGuard` + `@CurrentUser()`. Routes: `POST /meetings` (201), `GET /meetings` (200, paginated `?page&limit` defaults 1/20, ordered `startTime asc, id asc` — **the `id` tie-breaker is load-bearing for OFFSET paging; an e2e test pins it**), `GET /meetings/:id` (200), `PATCH /meetings/:id` (200, real partial update — omitted field left alone, `description: null` clears it; the `endTime > startTime` cross-field check lives in the handler because it needs the stored value), `DELETE /meetings/:id` (204), `DELETE /meetings/:meetingId/files/:fileId` (204). Deletes remove **row then bytes** (via `MeetingFileStorage.removeAll`, best-effort). All reads filter by the token's `userId`; unowned/missing → 404 (no existence leak, `P2025` → 404 too).
- **Meeting files (`src/meetings/meeting-files.controller.ts`, phases 1–3 of 5 — API done, UI not)** — `POST /meetings/:meetingId/files` (writes bytes to disk + metadata to `MeetingFile`), `GET …/files` (list), `GET …/files/:fileId` (streams as `StreamableFile`, RFC 5987 `filename*` for Cyrillic names). **Read `docs/meeting-file-upload-research.md` before extending** — it records decisions that are wrong-by-default vs the framework docs. Key facts held by that doc + e2e tests: ownership is checked in `MeetingOwnerGuard` (guards run before interceptors, so a non-owner can't write 100 MB before the 404); multer uses `diskStorage` + `randomUUID()` filenames; the 100 MB cap is `limits.fileSize` (= `MAX_UPLOAD_BYTES + 1`, the `+1` is load-bearing — do not "simplify" it) and the type check is a `fileFilter` (extension-keyed; mimetype allowlist is hygiene, not security). `GetMeetingFileHandler` filters on **both** `id` and `meetingId`. `UploadSizeLimitFilter` rewrites multer's bare 413 to name the limit, bound to the upload route only. **Phases 4–5 (frontend) do not exist yet.**
- **Error message language**: end-user-facing messages are **Russian** (product copy, rendered verbatim by the frontend — `Файл слишком большой…`, `Этот email уже зарегистрирован`); developer-only / handled-by-UI messages stay **English** (`Meeting not found`, `Invalid credentials`). No i18n layer. Requirements: `docs/meeting-file-upload.md`, plan `docs/meeting-file-upload-plan.md`.

### Storage & database

- **File storage (`src/storage/`)**: `StorageModule` is `@Global()`, exports the `UPLOAD_DIR` token + `MeetingFileStorage`, which owns **every** path built under it (`pathFor` is private — go through `open`/`remove`, never `join(uploadDir, …)` elsewhere). `open(storedName, expectedSize)` `stat`s first and 404s unless the file exists, is regular, **and** matches size (a row can outlive its bytes; serving a mismatch aborts the client download). `resolveUploadDir` reads `UPLOAD_DIR` (relative → `process.cwd()`, default `./uploads`) and `mkdirSync`s it at startup. In Docker it must sit on a mounted volume. E2e isolates onto its own directory via `test/setup-e2e.ts` — do not "clean up" the redundant-looking isolation asserts in `meeting-files.e2e-spec.ts`.
- **E2e has its own database**: `test/setup-e2e.ts` rewrites `DATABASE_URL` to `…_e2e` before `ConfigModule` reads `.env` (dotenv leaves an already-set var alone); `test/global-setup.ts` runs `prisma migrate deploy` against it. The suites wipe `user`/`meeting` wholesale, so `assertE2eDatabase()` (`test/e2e-database.ts`) **refuses to run** unless the URL ends in `_e2e` — the isolation is worth exactly that check.
- **Test litter cleanup (`scripts/clean-e2e.ts`, `npm run db:clean-e2e`)**: (1) **browser-e2e accounts** land in the **dev** database/uploads (they drive the dev server); registered under an `e2e-` prefix, deleted with their files + avatars by Playwright's `globalTeardown`. (2) **Orphaned bytes** — a cascade removes rows but never files; the sweep is the only thing that reclaims them (skips anything <1h old, since multer writes before the row exists). It sweeps two spaces against two reference sets: top-level files vs `MeetingFile.storedName`, and `avatars/` vs `User.avatarUrl`.
- **Prisma (`src/prisma/`)**: `PrismaModule` is `@Global()`, exports `PrismaService`. `schema.prisma`: `User` (`email` unique, `passwordHash`, `name?`, `avatarUrl?`, `avatarColor` **required** — the default-avatar colour name from `AVATAR_COLOR_SOLUTIONS`, set at registration, stable thereafter), `Meeting`, `MeetingFile` (`storedName` unique, `meetingId` indexed). Both relations 1-to-many `onDelete: Cascade` (rows only — bytes on disk survive). Migrations in `prisma/migrations/`, applied via `npx prisma migrate dev`; local Postgres must be running for migrations/`dev`/`test:e2e`.
- **Prisma pinned to `^6.19.2`** (both packages) — deliberate: Prisma 7.x changes the `datasource`/`env()` syntax and requires `prisma.config.ts` + a driver adapter, incompatible with the current setup. Don't let `npm update` bump past 6.x; upgrading to 7 is its own deliberate migration.
