# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@video-meetings/api` — a NestJS 11 backend. Part of the `video-meetings` Turborepo monorepo; see the root `CLAUDE.md` for cross-workspace commands and architecture. Runs on port 3001 (`process.env.PORT` overrides).

## Commands

Run from this directory, or from the repo root with `-w @video-meetings/api`:

```bash
npm run dev            # nest start --watch
npm run build           # nest build -> dist/
npm run start            # node dist/main
npm run lint             # eslint "{src,test}/**/*.ts"
npm run lint:fix
npm run test              # jest (unit tests, *.spec.ts under src/)
npm run test:e2e           # jest --config ./test/jest-e2e.json (*.e2e-spec.ts under test/) — requires Postgres running (`npm run db:up` from repo root)
npm run clean               # rm -rf dist
npm run db:seed              # ts-node prisma/seed.ts — the fixed dev account (see below)
npx prisma migrate dev      # apply/create migrations against the running Postgres
```

**Fixed development account** (`prisma/seed.ts`): `test@mail.com` / `Pasword1!`, with three sample meetings so the account is worth opening. The seed is idempotent — it upserts the user and replaces that user's meetings, so re-running neither fails on the unique email nor piles up duplicates. **It does not survive `npm run test:e2e`**: the API e2e specs call `prisma.user.deleteMany()` in `beforeEach`, which takes this account with everything else, so re-seed after a test run. Nothing in the test suites depends on it — they mint their own throwaway users; it exists only for poking at the app by hand. The password is written in the seed and in this file on purpose: it guards nothing but a local dev database, and must never be reused anywhere real.

Run a single test:

```bash
npx jest app.controller.spec.ts
npx jest -t 'should return a healthy response'
```

## Architecture

- Standard NestJS module/controller/service structure: `src/app.module.ts` wires `AppController` + `AppService`; `src/main.ts` bootstraps the app with a CORS allowlist (origins from the `CORS_ORIGINS` env var, comma-separated, default `http://localhost:3000`) and `app.enableShutdownHooks()` (so Prisma's `onModuleDestroy` disconnect fires on SIGTERM/SIGINT).
- Consumes shared response types from `@video-meetings/shared` (`ApiResponse<T>`, `PaginatedResponse<T>`) resolved via the `paths` alias in `tsconfig.json` — no compiled artifact required, imports go straight to `packages/shared/src`.
- `tsconfig.json` extends the root `tsconfig.base.json` and adds NestJS-specific compiler options (`emitDecoratorMetadata`, `experimentalDecorators`, `module`/`moduleResolution: nodenext`). `tsconfig.build.json` extends `tsconfig.json` and additionally excludes `test/` and `*.spec.ts` files from production builds.
- Jest config lives inline in `package.json` (`rootDir: "src"`, `testRegex: ".*\\.spec\\.ts$"`) for unit tests; e2e tests use the separate `test/jest-e2e.json` config and live under `test/`.
- `.eslintrc.js` sets `root: true` (does not inherit the repo-root ESLint config) and layers `plugin:prettier/recommended` on top of the `@typescript-eslint/recommended` rules.
- **CQRS**: `apps/api` uses CQRS (`@nestjs/cqrs`) for its feature modules — see **`docs/architecture/cqrs.md`** for the full guide: Command-vs-Query rule, folder/naming conventions, error propagation through the buses, and the step-by-step recipe for adding a new command or query. Read it before adding an operation.
- **Auth (`src/auth/`)**: Uses CQRS via `@nestjs/cqrs`. `AuthController` dispatches `RegisterCommand` and `LoginCommand` through `CommandBus`; the logic lives in `@CommandHandler`s under `src/auth/commands/handlers/` (`RegisterHandler`, `LoginHandler`). JWT issuance is centralized in `TokenService` (`src/auth/token.service.ts`), injected by both handlers. `AuthModule` imports `CqrsModule` and registers the two handlers and `TokenService` as providers, alongside `JwtModule.registerAsync` configured from `ConfigService` (secret/TTL). Login is modeled as a Command (not a Query); domain events (`EventBus`) are intentionally deferred with no consumer yet. `POST /auth/register` (201) and `POST /auth/login` (200, `@HttpCode(200)`) both validate `RegisterUserDto`/`LoginDto` (`class-validator`) and return `ApiResponse<{ accessToken, user: { id, email } }>` from `@video-meetings/shared`, wrapped manually in the controller (`{ success, message, data }`). Passwords are hashed with the native `bcrypt` package (10 rounds); JWT payload is `{ sub, email }`. Errors use Nest's default exception shape: 400 (validation), 401 with the same `'Invalid credentials'` message for both unknown email and wrong password, 409 for a duplicate email on register. HTTP behavior is unchanged from the pre-CQRS version—the existing e2e suite covers it. See `docs/superpowers/specs/2026-07-14-auth-cqrs-refactor-design.md`. `AuthModule` also provides and **exports** a reusable `JwtAuthGuard` (`src/auth/guards/`, verifies the `Authorization: Bearer` token with `JWT_SECRET` and attaches `{ userId, email }` to the request) plus a `@CurrentUser()` param decorator (`src/auth/decorators/`) and the `AuthUser` type — the JWT is now verified on incoming requests, not only issued. **Auth no longer touches the database directly**: `RegisterHandler` hashes the password then dispatches `CreateUserCommand`, and `LoginHandler` dispatches `GetUserByEmailQuery` — both handled by the Users module over the shared CQRS bus (`RegisterHandler` injects `CommandBus`, `LoginHandler` injects `QueryBus`; neither injects `PrismaService`). Password hashing/verification and the timing-attack dummy-hash stay in Auth; see the cross-module dispatch section in `docs/architecture/cqrs.md` and `docs/superpowers/specs/2026-07-15-auth-users-split-design.md`.
- **Users (`src/users/`)**: owns all access to the `user` table; the only place `prisma.user` is touched. Exposes its operations solely as CQRS handlers (no controller yet — user-facing HTTP like profile comes later): `CreateUserCommand(email, passwordHash)` → `CreateUserHandler` (`prisma.user.create`; a `P2002` unique-email violation becomes the `409 { message: 'Этот email уже зарегистрирован', field: 'email' }` conflict, moved here from auth since this is where the write happens) and `GetUserByEmailQuery(email)` → `GetUserByEmailHandler` (`prisma.user.findUnique`, returns the full `User` including `passwordHash` as an internal cross-handler contract — never serialized to HTTP). `UsersModule` imports `CqrsModule` and registers the two handlers as providers; it has no controller and no exports, and other modules reach it only through the bus (Auth depends on the command/query classes as value objects, not on `UsersModule`). `CreateUserCommand` takes a ready `passwordHash` — Users is unaware of bcrypt.
- **Meetings (`src/meetings/`)**: CQRS module for per-user meetings, the first protected routes. `MeetingsController` is guarded by `@UseGuards(JwtAuthGuard)` (imports `AuthModule` for the exported guard + `JwtModule`) and reads the owner via `@CurrentUser()`. `POST /meetings` (201) dispatches `CreateMeetingCommand` (`@CommandHandler`); `GET /meetings` (200) dispatches `ListMeetingsQuery` (`@QueryHandler`, paginated `?page&limit`, defaults 1/20, ordered `startTime asc` **then `id asc`**) and returns `PaginatedResponse<Meeting>`; the `id` tie-breaker is what makes `skip`/`take` safe and must not be dropped — meetings sharing a `startTime` (a recurring slot, a bulk import) have no defined order without it, so Postgres may order them differently per query and OFFSET paging then serves one row on two pages while another is never returned at all (measured on 80 meetings over 28 distinct start times: 2 duplicated, 2 unreachable). An e2e test pins it; `GET /meetings/:id` (200) dispatches `GetMeetingQuery`. `PATCH /meetings/:id` (200) dispatches `UpdateMeetingCommand` and is a real partial update: an omitted field is left alone (Prisma skips an `undefined` column), while `description: null` **clears** it — `@IsOptional()` waves `null` through, which is what makes the two distinguishable. `UpdateMeetingHandler` reads the row before writing, for two reasons that are both load-bearing: it establishes ownership (unowned reads as missing), and the «endTime after startTime» rule needs the **stored** value for whichever side the request omitted — moving only a meeting's end must still land after the start already in the database. No DTO-level rule can see that, which is why this one check lives in the handler while everything else stays in `UpdateMeetingDto`. All queries filter by the token's `userId`; an unowned/missing meeting → `404` (no existence leak). Create validates `endTime > startTime` → `400`. Handlers inject `PrismaService` and return the Prisma `Meeting` type. The module also registers `MulterModule.registerAsync` (disk storage, UUID filenames) and `MeetingFilesController` — see the meeting file upload bullet below.
- **Meeting files (`src/meetings/meeting-files.controller.ts`, phases 1–3 of 5 done — the API is complete, the UI is not)**: `POST /meetings/:meetingId/files` accepts one multipart `file` part, writes the bytes to disk and the metadata to `MeetingFile`; `GET /meetings/:meetingId/files` lists that meeting's metadata (`createdAt asc`, `id` as tie-breaker, empty array when there are none); `GET /meetings/:meetingId/files/:fileId` streams the bytes back as a `StreamableFile`. All three are guarded at the class level, so ownership is enforced once for every route. Download sets `Content-Disposition: attachment; filename*=UTF-8''…` (RFC 5987 — a bare `filename="…"` is ASCII-only and would mangle a Cyrillic name) plus `Content-Type`/`Content-Length` from the row. `GetMeetingFileHandler` filters on **both** `id` and `meetingId`: the guard only proves the caller owns the meeting in the path, so without the second filter a file of another meeting could be fetched by id — including another meeting the same user owns. **Read `docs/meeting-file-upload-research.md` before extending this** — it is the technical ground truth and records decisions that are wrong-by-default if you follow the framework docs literally. Already applied here: the controller is guarded by `@UseGuards(JwtAuthGuard, MeetingOwnerGuard)` and `MeetingOwnerGuard` (`src/meetings/guards/`) does the owner lookup **in a guard, not the handler**, because guards run before interceptors — a handler-level check would let a non-owner write 100 MB before their 404 (an unowned or missing meeting both give `404 Meeting not found`); multer uses `diskStorage` with a `randomUUID()` `filename`, since the client-supplied `originalname` can carry `../`; the controller repairs `originalname` via `decodeOriginalName` (`src/meetings/decode-original-name.ts`, unit-tested) so Cyrillic names survive busboy's latin1 decode — note this is deliberately **conditional**, since an unconditional latin1→UTF-8 round-trip corrupts names from clients that send RFC 5987 `filename*`; and `UploadMeetingFileHandler` `unlink`s the file if the Prisma write fails, since multer writes before the handler runs and a post-write failure would otherwise orphan it. Cleanup on a **client disconnect needs no code of ours** — multer 2.2.0 hooks `aborted`/`error`/`close` itself and removes both finished and partial files (verified in `node_modules/multer/lib/make-middleware.js`); re-verify if multer is upgraded. `MeetingFileResponse` (`src/meetings/meeting-file.response.ts`) is what goes over HTTP — it drops `storedName`, which stays server-side. **Upload validation** lives in `src/meetings/meeting-file-validation.ts` (unit-tested), wired into `MulterModule` in `meetings.module.ts`: the 100 MB cap is `limits.fileSize` — **not** a `ParseFilePipe` validator, which would only run once all 101 MB were already on disk — and the type check is a `fileFilter`, which rejects before a byte is written. What multer gets is `MULTER_FILE_SIZE_LIMIT` = `MAX_UPLOAD_BYTES + 1`, and the `+ 1` is load-bearing, not a fudge: busboy trips when the byte count **reaches** `fileSize` (`if (fileSize === fileSizeLimit)` in `busboy/lib/types/multipart.js`), so passing the cap verbatim rejects a file of exactly 100 MB while the 413 text promises 100 MB is fine. Verified before and after, and an e2e test pins the boundary — do not "simplify" the `+ 1` away. Two non-obvious things hold it together. First, Nest maps `LIMIT_FILE_SIZE` to `PayloadTooLargeException` carrying multer's bare `File too large`, which never names the limit the PRD requires; `UploadSizeLimitFilter` (`src/meetings/filters/`) rewrites that body and is bound with `@UseFilters` to the upload route **only**, since app-wide it would mislabel unrelated body-size errors. Second, `transformException` in `@nestjs/platform-express` passes an `HttpException` through untouched, which is why `fileFilter` can hand its own `BadRequestException` to the callback and have the message survive verbatim. The filter keys on **extension** (`.mp3/.wav/.m4a/.mp4`); the mimetype allowlist is deliberately broad (clients disagree, and `application/octet-stream` is common) — it is hygiene, not security, since both extension and mimetype are client-written and neither inspects the bytes. That is the accepted compromise: antivirus is out of PRD scope, uploads are never executed, only the owner reads them. Rejections leave nothing behind (row, partial file, or earlier uploads) — multer removes what it wrote, covered by e2e. **Still to come: phases 4–5, the frontend — do not assume they exist.**
- **Language of error messages**: messages the end user reads are **Russian** (`Файл слишком большой…`, `Тип файла не поддерживается…`, `Этот email уже зарегистрирован`) — the frontend renders them verbatim, so they are product copy, not diagnostics. Messages that only ever reach a developer or a 404/401 the UI handles itself stay **English** (`Meeting not found`, `File not found`, `Invalid credentials`). There is no i18n layer; if a second language is ever needed these become codes. Requirements live in `docs/meeting-file-upload.md`, the phased plan in `docs/meeting-file-upload-plan.md`, and the backlog is milestones 1–5 in GitHub issues.
- **File storage (`src/storage/`)**: `StorageModule` is `@Global()` and exports the `UPLOAD_DIR` string token — the resolved absolute directory uploads are written to — plus `MeetingFileStorage` (`meeting-file-storage.service.ts`), which owns **every** path built under it — `pathFor` is private on purpose, so the only ways in are `open`/`remove`; do not `join(uploadDir, …)` anywhere else. `open(storedName, expectedSize)` `stat`s the file first and 404s unless it exists, is a regular file, **and** matches `expectedSize`. All three checks are load-bearing, since a row can outlive its bytes whole or in part (wiped directory, DB dump restored beside a stale volume, interrupted copy): without the existence check `createReadStream` raises ENOENT _inside_ the stream once 200 is on the wire; without the size check the `Content-Length` taken from the row promises bytes the stream cannot deliver, and the client's download **aborts** instead of failing cleanly (verified by truncating a file on disk — the request died with `Error: aborted`). Stored files are immutable, so a size mismatch can only be corruption, and serving it would hand back a silently broken file while reporting success. `resolveUploadDir` reads the `UPLOAD_DIR` env var (relative paths resolve against `process.cwd()`, default `./uploads`) and `mkdirSync`s it at startup, because multer does not create its own destination. `apps/api/uploads` is git-ignored. The e2e run is isolated onto its own directory by `test/setup-e2e.ts` (wired via `setupFiles` in `test/jest-e2e.json`), which sets `process.env.UPLOAD_DIR` to `test/.uploads-e2e` before the app loads — dotenv never overrides an already-set `process.env` var, so this wins over `.env`. `meeting-files.e2e-spec.ts` wipes that directory between tests, so it both imports `setup-e2e` directly (the import's side effect isolates it even without `setupFiles`) and asserts the resolved dir is the isolated one before running — do not "clean up" either as redundant. In Docker the directory must sit on a mounted volume or uploads die with the container.
- **Prisma / database (`src/prisma/`)**: `PrismaModule` is `@Global()` and exports `PrismaService` (extends `PrismaClient`, connects/disconnects on module init/destroy) so any module can inject it without re-importing. `prisma/schema.prisma` has `User` (`id`, `email` unique, `passwordHash`, `createdAt`), `Meeting` (`id`, `title`, `description?`, `startTime`, `endTime`, `createdAt`, `userId`) and `MeetingFile` (`id`, `originalName`, `storedName` unique — the UUID on disk, `size`, `mimeType`, `createdAt`, `meetingId` indexed). Both relations are 1-to-many with `onDelete: Cascade`: `User.meetings` ↔ `Meeting.user` and `Meeting.files` ↔ `MeetingFile.meeting`. Note the cascade only removes rows — the bytes on disk survive a deleted meeting; disk cleanup belongs to the meeting-management feature, not here. Migrations are in `prisma/migrations/` and applied/created with `npx prisma migrate dev`. The local Postgres (`docker-compose.yml` at repo root, `npm run db:up`) must be running for migrations, `dev`, and `test:e2e`.
- **Config (`main.ts`, `app.module.ts`)**: `ConfigModule.forRoot({ isGlobal: true })` reads `apps/api/.env` (see `.env.example`: `DATABASE_URL`, `JWT_SECRET`, `JWT_EXPIRES_IN` default `1h`, `CORS_ORIGINS` default `http://localhost:3000`, `UPLOAD_DIR` default `./uploads`). `main.ts` also installs a global `ValidationPipe({ whitelist: true, transform: true })`.
- **Prisma pinned to `^6.19.2`** (both `prisma` and `@prisma/client`) — deliberate, not stale: Prisma 7.x changes the `datasource`/`env()` schema syntax and requires a `prisma.config.ts` plus a driver adapter, which is incompatible with the current `schema.prisma`/migration setup. Do not let a routine `npm update` bump past 6.x; upgrading to 7 needs its own deliberate migration.
