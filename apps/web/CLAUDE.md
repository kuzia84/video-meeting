# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@video-meetings/web` — Next.js 15 (App Router, React 19) frontend, Turbopack in dev. Part of the `video-meetings` monorepo (see root `CLAUDE.md` for cross-workspace commands/architecture). Port 3000 (API on 3001).

## Commands

Scripts are in this workspace's `package.json`; run from here or from the root with `-w @video-meetings/web`. `npm run dev` uses Turbopack. The only one with non-obvious behavior is `npm run test:e2e` (Playwright, browser e2e in `e2e/`):

**Browser e2e** (`e2e/`, `playwright.config.ts`) run the real stack (frontend → real API → real Postgres, `npm run db:up` first). Playwright starts both dev servers itself and reuses a running `npm run dev`. Load-bearing config: the API readiness URL is `/health` (not `/` — the API 404s there, which would defeat `reuseExistingServer` and cause EADDRINUSE); `workers: 1` (shared database); `use.timezoneId = 'UTC'` (the app formats dates in the browser zone, so without a pinned zone machines disagree). Not wired into turbo's `test` task (keeps `npm run test` fast/dependency-free).

These tests write to the **dev** database/uploads (they drive the dev server), so they clean up: `globalTeardown` runs `npm run db:clean-e2e -w @video-meetings/api`, matching the **`e2e-` prefix** in `registerUser` (keep it) — never fails the run. They **seed through the API**, not the UI (`e2e/support.ts`: `registerUser`, `createMeeting(s)`, `signIn`) — the creation form is tested separately; `signIn` writes the token into `sessionStorage` via `addInitScript`. Every run registers a unique email (shared dev database).

Note: this is committed unattended tests, **not** the Playwright **MCP** server from `docs/architecture/frontend-ui.md` (which drives a browser by hand to eyeball layout). Both exist on purpose.

## Architecture

App Router under `src/app/` (`layout.tsx`, `page.tsx`, route folders). Path aliases in `tsconfig.json`: `@/*` → `src/*`; `@video-meetings/shared` → `packages/shared/src` (source, not `dist/`, via `transpilePackages` in `next.config.ts`), so shared type/runtime changes need no build step (unlike the API — see root `CLAUDE.md`). `tsconfig.json` extends the root base with Next overrides (`ESNext`/`bundler`, `jsx: preserve`, `noEmit`). `.eslintrc.js` sets `root: true`, extends `next/core-web-vitals` + `next/typescript`.

### Cross-cutting conventions

- **Auth-guarded resources travel as a header, never a URL.** The token lives in `sessionStorage` (invisible to the server — no `middleware.ts`), so JWT-guarded routes cannot be reached by `<img src>` / `<a href>`. Fetch with the `Authorization` header instead: JSON via the API client (below); binary (avatar, file download) via `fetch → blob() → object URL`, revoked on unmount/next-tick. **Route protection is client-side**: each protected page checks `getAccessToken()` in a mount effect and `router.replace('/login')` when absent (and on a `401`, after clearing the dead token). New protected pages follow this same in-effect guard.
- **API client (`src/lib/api/client.ts`)**: `fetchJson<T>(path, options)` is the one shared `fetch` wrapper (no HTTP library) — import it, don't inline `fetch`. Resolves the base URL from `NEXT_PUBLIC_API_URL` (default `http://localhost:3001`), guards `res.json()` against non-JSON bodies, and on non-2xx throws `ApiError` (`status`, `messages: string[]` normalizing the API's two error shapes, plus optional `field` — **branch on `err.field`, not `err.status` alone**, to target a form field). `fetchPaginated` is a sibling returning the full `data`/`total`/`page`/`limit` envelope. `fetchVoid` for 204s. Feature files (`auth.ts`, `meetings.ts`, `profile.ts`, `meeting-files.ts`) validate their own expected shape. `src/lib/auth/token.ts` is the **only** place a token is persisted (`saveAccessToken`/`getAccessToken`/`removeAccessToken`, SSR-guarded; `saveAccessToken` throws a distinct `StorageError`; `getUserEmailFromToken` decodes the JWT client-side for display only).

### Shared providers & components (use these; don't hand-roll)

- **`CurrentUserProvider` / `useCurrentUser` (`src/lib/current-user/current-user-context.tsx`)** — the **single source of the signed-in user**. Mounted at the app root (inside `Providers`), stays mounted across client navigations, exposes `{ user, status, errorMessage, reload, setUser }` (`status`: `loading | ready | error | unauthenticated`). One `GET /users/me` serves both the header chip and the profile page; pushing a rename through `setUser` updates every consumer with no reload. Reconciles on every route change: (re)fetches when the token **first appears** (login is a client `router.push`, not a reload) and resets to `unauthenticated` on logout. It does **not** redirect or clear the token on a `401` — it only reports `unauthenticated`; the consuming page decides. `useCurrentUser()` throws outside the provider.
- **`AppHeader` (`src/components/app-header.tsx`)** — the header every signed-in page wears (logo home-link, user chip, logout). Use it, don't hand-roll (it was extracted after copies drifted). The chip is a `UserAvatar` + display name wrapping a `NextLink` to `/profile` (`aria-label` contains the visible name for WCAG 2.5.3). Reads from `useCurrentUser`, so a rename updates it live. With no user it's a bare logo + logout (the page owns auth outcomes).
- **`PageShell` (`src/components/page-shell.tsx`)** — the frame for a signed-in **content** page: `AppHeader` above a centered `max-w-3xl` main. Wrap new content pages in it. Bare loading/redirect spinners stay outside it; the home view inlines its own layout (pre-existing exception).
- **`UserAvatar` (`src/components/user-avatar.tsx`, client)** — renders the uploaded picture (blob from JWT-guarded `GET /users/me/avatar` via `fetchAvatarBlob` + object URL, keyed on `avatarUrl`) else the `DefaultAvatar` letter circle. Decorative (`aria-hidden`, `alt=""`). Used by the header chip and profile card.
- **`DefaultAvatar` (`src/components/default-avatar.tsx`)** — the letter circle: `avatarInitial(name, email)` on the user's own colour (`findAvatarColorSolution` keyed by `UserProfile.avatarColor`). **Theme-aware with no JS** — sets the four palette colours as inline CSS vars consumed by `.avatar-default` in `globals.css`, dark variant under `[data-theme='dark']` (no first-render flash; both themes clear WCAG AA, verified in shared unit tests). Decorative; the only web consumer of the shared runtime avatar helpers.
- **`MeetingForm` (`src/components/meeting-form.tsx`)** — meeting fields shared by create + edit (same rules, one place). Owns validation + per-field errors; `onSubmit` rejecting is how a caller reports failure. `children` is a slot between fields and buttons (the create page's file picker — files belong to creating, not editing).
- **`ConfirmDeleteDialog` (`src/components/confirm-delete-dialog.tsx`)** — both deletions go behind it (irreversible, takes bytes). The confirm button has **no** `slot="close"` (stays open until the request settles, so a failure has somewhere to report). Give trigger and confirm **different** names.

### Pages

- **`/register`, `/login`** — public (server `page.tsx` renders logo/tagline + a `'use client'` form; mirror one for new auth screens).
- **`/` (home, `home-view.tsx`)** — greeting (email from the JWT), meeting count, the full meeting list a page at a time (10/page, HeroUI `Pagination`, hidden when it fits one page) or an empty state, a "Создать встречу" CTA → `/meetings/new`, logout. Order is the API's `startTime asc`. Details: the `<ul>` carries `aria-label="Список встреч"` (HeroUI renders its own `<li>`s); the CTA is **hidden while the empty state shows** (the empty state has its own invitation — one primary per context) but stays on a load error; page links windowed by `paginationRange` (`src/app/pagination-range.ts`, capped at 7 slots); `loadPage` stamps requests and ignores superseded responses; a paging failure keeps list + pagination on screen.
- **`/profile` (`profile-view.tsx`)** — wears `AppHeader`, reads from `useCurrentUser` (not its own fetch). Redirects to `/login` when the source resolves `unauthenticated`; a load error shows a «Попробовать снова» calling `reload()`. Renders a card: `DefaultAvatar`/`UserAvatar` + display name (`name`, else email, with a «Имя пока не задано» hint) + a **read-only** email `TextField` (never editable). Sub-forms:
  - **`ProfileNameForm`** — HeroUI `Form` (`validationBehavior="aria"`, so the submit-handler length check is load-bearing, mirroring the API's 1..100), `PATCH`es via `updateProfileName()`, hands the returned `UserProfile` to `setUser` (card + avatar + header refresh, no reload). A `401` bubbles via `onUnauthorized`; other failures show verbatim.
  - **`AvatarUpload`** — native `<input type="file">` styled as a button (`sr-only` input + `focus-within` ring — keeps keyboard access), `accept` lists exactly what the API takes (no `image/*` wildcard). Uploads immediately via `uploadAvatar()` (multipart, **auth header without `Content-Type`** so the browser sets the boundary). Success → `setUser`; rejections show **verbatim from the API**, previous avatar stays.
  - **`PasswordChangeForm`** — current/new/confirm fields. Confirmation-match and new-password length checked **client-side** (`PASSWORD_MIN=8` hand-synced with the backend `IsPassword`), then `changePassword()` (`POST /auth/change-password`, 204). Success clears the fields; a wrong current password comes back `400` with `field: 'currentPassword'` and is pinned to that input.
- **`/meetings/new` (`create-meeting-view.tsx`)** — title/description/start/end + optional file picker, then `router.push` to the created meeting. **Order forced**: meeting created first (files need its id), so a failed upload goes to its own notice with a link to the meeting (**not** the form error slot, which would read as "not created"). Validation lives in `handleSubmit` and is **load-bearing** — `validationBehavior="aria"` does not block submission (mutation-tested: delete the checks and four e2e fail). Errors go into a `fieldErrors` record fed to `<Form validationErrors>`; each field clears its own error in `onChange`. `datetime-local` → ISO for the API.
- **`/meetings/[id]` (`meeting-view.tsx`)** — server `page.tsx` awaits `params` (Next 15 Promise), client view fetches. Four states — `loading`, `ready`, **`missing`**, `error` — and `missing` is **not** an error (the API returns the same 404 for someone else's meeting as for a nonexistent one, so the page can't tell them apart). Auth gated in a mount effect.
  - **Meeting files block (`meeting-files.tsx`)** — rendered below the details once the meeting resolved; owns its own loading/error state **separately** from the meeting and does **not** redirect on `401`/`404` (the meeting view owns the page-level outcome). Download via `downloadMeetingFile` (blob + object URL, revoke next tick; `link.download` carries the filename).
  - **Upload (`uploadMeetingFile` in `src/lib/api/meeting-files.ts`)** — the one call built on **XMLHttpRequest** (not the shared `fetch` wrapper) because `fetch` has no upload-progress events. Still takes base-URL/error-mapping from `client.ts`. **Never set `Content-Type` on the FormData request** (only the browser knows the multipart boundary). Rejections shown **verbatim from the API**. On success the created row is appended from the response. (E2e throttles upload to 200 KB/s via CDP so the progress bar is testable.)

### UI

**HeroUI v3 + Playwright MCP — read `docs/architecture/frontend-ui.md`** for the full guide: HeroUI v3 setup/conventions (compound components, `onPress`, semantic variants, `next-themes` for light/dark — **no `HeroUIProvider` in v3**), the `validationBehavior="aria"` gotcha for any `Form`, and the Playwright MCP workflow. Wiring: `postcss.config.mjs`, `globals.css` (`@import 'tailwindcss'` then `@import '@heroui/styles'`), `src/app/providers.tsx` (`next-themes` `ThemeProvider`, no `HeroUIProvider`), `layout.tsx` imports `globals.css` + wraps in `<Providers>`.

## UI change verification (mandatory)

Any change affecting rendered UI is not done until visually verified **via Playwright MCP** (no other tool). Apply the **UI UX Pro Max** skill when designing/reviewing. The dev server is already running — don't start it yourself.

## Keeping documentation current

When a change alters this app's architecture — new routing/layout, new shared-package usage, changed build config or path aliases — update this file (and root `CLAUDE.md` if monorepo-wide) in the same change.
