# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@video-meetings/web` — a Next.js 15 (App Router, React 19) frontend, using Turbopack in dev. Part of the `video-meetings` Turborepo monorepo; see the root `CLAUDE.md` for cross-workspace commands and architecture. Runs on the Next.js default port 3000 (the API runs on 3001).

## Commands

Run from this directory, or from the repo root with `-w @video-meetings/web`:

```bash
npm run dev        # next dev --turbopack
npm run build        # next build
npm run start          # next start
npm run lint             # next lint
npm run lint:fix
npm run test:e2e          # playwright test — browser e2e in e2e/
npm run clean               # rm -rf .next
```

Browser e2e (`e2e/`, `playwright.config.ts`) run the real stack: the frontend against the real API against the real Postgres (`npm run db:up` from the repo root must have been run). `playwright.config.ts` starts both dev servers itself and reuses them when they are already up — so the usual `npm run dev` session is enough. Two things there are load-bearing: the API's readiness URL is `/health`, **not** `/` (the API has no root route and 404s there, which reads as "not up", so `reuseExistingServer` would miss a running API and try to start a second one on the same port — EADDRINUSE); and `workers: 1`, since the suite shares one database.

`test:e2e` is deliberately **not** wired into turbo's `test` task (same as the API's own e2e): `npm run test` at the root stays fast and dependency-free, while browser tests need Postgres and installed browsers (`npx playwright install chromium`).

`use.timezoneId` is pinned to `UTC` so assertions on rendered dates mean the same thing on every machine — the app formats times in the browser's zone, so without it a developer in UTC+3 and a CI box in UTC disagree about what a given instant reads as, and whoever wrote the test wins by accident.

These tests write to the **dev** database and upload directory — they drive the dev server, which reads its own `.env`, so they cannot be given their own the way the API's e2e are. Instead they clean up: `globalTeardown` runs `npm run db:clean-e2e -w @video-meetings/api`, which deletes every `e2e-`-prefixed account and unlinks its uploads (a cascade would leave the bytes). It never fails the run — housekeeping must not turn a green suite red. Keep the `e2e-` prefix in `registerUser`: it is what the cleanup matches on.

Tests seed through the API rather than the UI (`e2e/support.ts`: `registerUser`, `createMeeting`/`createMeetings`, `signIn`) — the creation form does not exist yet, and driving it would test that form instead of the screen under test. `signIn` writes the token into `sessionStorage` via `addInitScript`, mirroring what the app reads. Every run registers a **unique** email, because the suite shares the dev database with everything else.

This is not the Playwright **MCP** server from `docs/architecture/frontend-ui.md`: that drives a browser by hand to eyeball layout, this is committed tests that run unattended. Both exist, on purpose.

## Architecture

- App Router structure under `src/app/` (`layout.tsx`, `page.tsx`, plus route folders like `src/app/register/`, `src/app/login/`).
- **Pages & auth gating**: `/register` and `/login` are public (server `page.tsx` renders the logo/tagline + a `'use client'` form — mirror one when adding auth screens). `/` is the authenticated home (`page.tsx` → `home-view.tsx`, `'use client'`): greeting (email decoded from the JWT), meeting count, the **full meeting list a page at a time** (10 per page, `Pagination` from HeroUI; the control is hidden when everything fits on one page) or an empty state, a "Создать встречу" CTA pointing at `/meetings/new` (that page arrives in a later phase — the CTA currently leads to a 404 by design), and logout. Order is the API's `startTime` ascending, the only order it offers. Details worth keeping: the meetings `<ul>` carries `aria-label="Список встреч"` because HeroUI's pagination renders `<li>`s of its own and the two are otherwise indistinguishable to assistive tech and to tests; the page-level CTA is **hidden while the empty state is showing**, since the empty state has its own invitation and two identical primary buttons would compete (HeroUI's own guidance: one `primary` per context) — on a load error it stays, so a failed load doesn't strand the user; page links are windowed by `paginationRange` (`src/app/pagination-range.ts`) rather than one link per page, which caps the control at 7 slots no matter how many pages exist; `loadPage` stamps each request and ignores responses from superseded ones, so clicking through pages fast cannot land the user on a page they did not ask for last; and a failure while paging keeps the list and pagination on screen and reports beside them, because only the very first load has nothing to fall back on. **Route protection is client-side** — the token lives in `sessionStorage` (invisible to the server, so there's no `middleware.ts`); the home view checks `getAccessToken()` in a mount effect and `router.replace('/login')` when absent (and again on a `401` from the API, after clearing the dead token). New protected pages follow this same in-effect guard. `/profile` (`page.tsx` → `profile-view.tsx`, `'use client'`) is such a page: it wears the `AppHeader` and reads the current user from the shared **`useCurrentUser`** source (below) rather than fetching itself — so the header chip and this page share one `GET /users/me`, and a rename here updates both at once. It redirects to `/login` when that source resolves to `unauthenticated` (no/expired token — the provider never redirects on anyone's behalf, the page does), showing the loading screen meanwhile so the page never flashes; a load error shows a «Попробовать снова» retry that calls the source's `reload()`, not a page reload. On success it renders a «Профиль» heading and a card whose header pairs a **`DefaultAvatar`** (see below) with the **display name** — `name` when set, otherwise the **email** (a blank/whitespace name counts as unset), with a «Имя пока не задано — показан email» hint under the fallback — plus the email in a **read-only** `TextField` (`isReadOnly`), which it stays in every phase since it identifies the account (the email is never editable). The **name is editable** here via **`ProfileNameForm`** (`src/app/profile/profile-name-form.tsx`, `'use client'`): a HeroUI `Form` (`validationBehavior="aria"`, so the empty/length check in the submit handler is load-bearing — mirroring the API's 1..100 rule) whose input is pre-filled with the current name, which `PATCH`es via `updateProfileName()` (`src/lib/api/profile.ts`) and, on success, hands the returned `UserProfile` back through `onSaved` — wired to the shared source's `setUser`, so the card title, the avatar letter here **and the header chip** all refresh without a reload, and `avatarColor` comes back unchanged so the circle keeps its colour on a rename. A «Имя сохранено» success alert confirms the save; a `401` bubbles to the page via `onUnauthorized` (`goToLogin` — clears the dead token, redirects to `/login`), every other failure shows verbatim (the API names the cause on a 400). The **avatar is uploaded** here via **`AvatarUpload`** (`src/app/profile/avatar-upload.tsx`, `'use client'`): a native `<input type="file">` styled as a button through its label (`sr-only` input + `buttonVariants`, `focus-within` ring — a fake button driving a hidden input loses keyboard access), whose `accept` lists exactly what the API takes (`.jpg,.jpeg,.png,.webp` + their mimetypes, no `image/*` wildcard that would offer a GIF only to be rejected). Picking a file uploads it immediately via `uploadAvatar()` (`src/lib/api/profile.ts` — multipart `POST /users/me/avatar`; it uses an **auth header without `Content-Type`** so the browser sets the multipart boundary itself) and, on success, hands the updated `UserProfile` to the shared source's `setUser` (so the new avatar propagates everywhere). Rejections (format/size/content) show **verbatim from the API** and the previous avatar stays put (the server swaps only on success); a `401` bubbles via `onUnauthorized`. The card and header render the avatar through **`UserAvatar`** (below), so a successful upload shows the picture — in place of the letter circle — in both at once. There is no shared `Meeting`-style local type here: the profile shape is the shared `UserProfile` from `@video-meetings/shared`, so `profile.ts` re-exports it for callers.
- **`DefaultAvatar` (`src/components/default-avatar.tsx`)**: the circle shown in place of an uploaded picture — the user's initial (`avatarInitial(name, email)` from `@video-meetings/shared`) on their own colour (`findAvatarColorSolution(colorName)` from the shared palette, keyed by `UserProfile.avatarColor`). It is **theme-aware without any JS theme detection**: it sets the solution's four colours as inline CSS variables (`--avatar-bg-light/-fg-light/-bg-dark/-fg-dark`) which the `.avatar-default` rule in `globals.css` consumes, picking the dark variant under the app's `[data-theme='dark']` selector — so there is no first-render flash and both themes clear WCAG AA contrast (verified in the shared palette's unit tests). It is **decorative** (`aria-hidden`), since the name/email is shown as text beside it. It is the fallback inside `UserAvatar` (below), never rendered directly by pages; the caller passes sizing via `className`. This is the only current consumer of the shared **runtime** avatar helpers on the web side — Next.js transpiles shared source, so importing runtime values here needs no build step (unlike the API — see root `CLAUDE.md`).
- **`UserAvatar` (`src/components/user-avatar.tsx`, `'use client'`)**: what pages actually render for a user's avatar — the **uploaded picture when there is one, else the `DefaultAvatar` letter circle**. The image sits behind the JWT-guarded `GET /users/me/avatar`, so it cannot be an `<img src>` to the endpoint (that sends no token); the bytes are fetched with the auth header (`fetchAvatarBlob`) and shown via an **object URL** (revoked on unmount/change), the same shape as the meeting-file download — and rendered with a plain `<img>` (a `blob:` URL is already in memory, so `next/image` can't optimize it; the `no-img-element` rule is disabled on that line with that reason). The fetch is keyed on `avatarUrl` (the stored name, a fresh UUID on every upload), so replacing the avatar re-loads the picture with no reload; while it loads, or when there is none, the letter circle stands in (never a blank gap). Decorative (`aria-hidden`, `alt=""`) — the name/email is text beside it. Used by both the header chip and the profile card, so an upload shows in both at once via the shared current-user source.
- **`AppHeader` (`src/components/app-header.tsx`)**: the header every signed-in page wears — logo linking home, the current-user chip, and logout. Use it; do not hand-roll a header. It exists because the same markup had been copied into home, the meeting page and the create form, and had already drifted (logout on two of the three, the logo a link on some), so each fix took three edits to stick. The **user chip** is a `UserAvatar` (the uploaded picture, else the letter circle) + display name (`name` when set, otherwise email) sitting next to the logout button, with the **avatar wrapping a `NextLink` to `/profile`** (`aria-label={`Профиль: ${displayName}`}` — the accessible name contains the visible name for WCAG 2.5.3 Label in Name) — clicking it opens the profile page. The user comes from the shared **`useCurrentUser`** source (below), **not** a fetch of its own, so a rename on the profile page updates this chip live, without a reload. When there is no user (loading, signed out, or a failed load) it stays a bare logo + logout, because the page around it owns auth outcomes (a 401 redirect, its own error UI) and the header must not race it.
- **`CurrentUserProvider` / `useCurrentUser` (`src/lib/current-user/current-user-context.tsx`)**: the **single source of the signed-in user** in the frontend. Mounted at the app root (inside `Providers`), it stays mounted across client navigations and exposes `{ user, status, errorMessage, reload, setUser }` (`status`: `loading | ready | error | unauthenticated`). Both the header chip and the profile page read from it, so one `GET /users/me` serves both, and pushing a rename through `setUser` updates every consumer at once — no reload. It reconciles on every route change (`usePathname`): it (re)fetches when the token **first appears** (login is a client `router.push('/')`, not a full reload, so a fetch-once-at-mount provider would miss it) and resets to `unauthenticated` when the token is gone (logout) — keyed on a `handledTokenRef` that skips a re-fetch when the auth state is unchanged, and, because the ref survives Strict Mode's mount→unmount→remount on the same instance, prevents a duplicate initial fetch. It does **not** redirect or clear the token on a `401` — it only reports `unauthenticated`; the consuming page decides (the profile page clears the dead token and routes to `/login`). Consume it via `useCurrentUser()` (throws outside the provider). The home view still decodes the greeting email from the JWT itself (`getUserEmailFromToken`) — a display-only concern independent of this source.
- **`PageShell` (`src/components/page-shell.tsx`)**: the frame a signed-in **content** page wears — the `AppHeader` above a centered `max-w-3xl` main column. The meeting view and the profile view both use it; wrap new content pages in it rather than re-declaring the same `div/main` wrapper (the two had an identical local `Shell`, the same copy-paste that `AppHeader` was pulled out to stop). Loading/redirect screens that render a bare centered spinner before the frame is warranted stay outside it — the home view, which inlines its own layout, is the pre-existing exception.
- **`MeetingForm` (`src/components/meeting-form.tsx`)**: the meeting fields, shared by creating and editing — the phase's own wording is «the same rules as when creating», and two copies would be two places for those rules to drift. It owns validation and per-field errors; `onSubmit` rejecting is how a caller reports a failure, which the form shows above the fields. `children` is a slot between the fields and the buttons (the create page's file picker) — files belong to creating, not editing, so they are not a prop on this component.
- **Create meeting (`/meetings/new`)**: `page.tsx` → `create-meeting-view.tsx` (`'use client'`) — title, description, start and end, an optional file picker, then `router.push` to the created meeting's page. **Order is forced**: files need a meeting id, so the meeting is created first and the files follow. Everything after that point keeps the meeting reachable, because it is already real — a failed upload therefore does **not** go to the form's error slot (which sits by the fields and would read as "the meeting was not created", the opposite of true) but to its own notice with a link to the meeting. This is the target the home page's CTA has pointed at since the list phase, which 404'd until now. Validation lives in `handleSubmit`, and it is **load-bearing, not belt-and-braces**: `validationBehavior="aria"` (mandatory for any `Form` here — see `docs/architecture/frontend-ui.md`) does **not block submission**, so the handler runs with empty and invalid fields. Mutation-tested: delete the checks and four e2e fail. Errors go into a `fieldErrors` record fed to `<Form validationErrors>`, so each reason renders under the field it belongs to rather than in one form-wide banner; each field clears its own error in `onChange`, because HeroUI leaves a submit-time error in place after the value changes. `datetime-local` yields a zone-less `YYYY-MM-DDTHH:mm`, converted to ISO for the API. The form renders only after the token check, so nobody about to be bounced to `/login` sees it flash first.
- **`ConfirmDeleteDialog` (`src/components/confirm-delete-dialog.tsx`)**: both deletions go behind it — each is irreversible and takes bytes with it. The confirm button deliberately has **no** `slot="close"`: it stays open until the request settles, so a failure has somewhere to be reported instead of vanishing into a closed dialog that implies success. Give the trigger and the confirm **different** names; they are different actions, and one name for both is one utterance with two outcomes.
- **Meeting files block (`src/app/meetings/[id]/meeting-files.tsx`)**: rendered on the meeting page below the details, once the meeting itself resolved — on a 404 there is no meeting to hang files off. Lists name, size and upload date, uploads, has its own empty state, and owns its own loading/error state **separately from the meeting**: a failed file list must not take the meeting page down with it, and the block deliberately does **not** redirect on `401`/`404` — the meeting view around it owns the page-level outcome, and two components racing to redirect on the same response would fight. Download is `downloadMeetingFile` in `src/lib/api/meeting-files.ts`: **not** an `<a href>` to the endpoint, because the route is JWT-guarded and the token lives in sessionStorage, so it can only travel as a header — a plain link sends no `Authorization` and gets a 401. Instead: fetch with the header → `blob()` → `URL.createObjectURL` → synthetic click → revoke on the **next tick** (revoking in the same task can cancel the download the click only just started). The consequence is that the whole file, up to the API's 100 MB cap, sits in the tab's memory; the alternative is short-lived signed URLs, a mechanism neither the PRD nor the current auth has. `link.download` carries the filename because a `blob:` URL has no `Content-Disposition` behind it.
- **Upload (`uploadMeetingFile` in `src/lib/api/meeting-files.ts`)**: the one call in the app built on **XMLHttpRequest** instead of the shared `fetch` wrapper, and not by preference — `fetch` has **no upload progress events at all** (only download), and the streaming workaround (`ReadableStream` body + `duplex: 'half'`) needs HTTP/2 and is unsupported in Safari and Firefox. It still takes base-URL resolution (`apiUrl`) and error mapping (`apiErrorFromText`) from `client.ts`, so a 401 here is the same `ApiError` as anywhere else. **Never set `Content-Type` on the FormData request**: only the browser knows the multipart boundary, and setting it by hand produces a body the server cannot parse. Rejections (type, 100 MB limit) are shown **verbatim from the API** — the PRD asks for the cause, and the API is what owns that wording. The picker is a real `<input type="file">` inside a `<label>` styled with `buttonVariants` (`sr-only` input): a fake button driving a hidden input loses keyboard access unless rewired by hand. The input's `value` is cleared after each attempt so re-picking the same file still fires `change`. On success the created row is appended from the response rather than re-fetching the list.
  **Testing progress**: the e2e throttles upload to 200 KB/s via CDP `Network.emulateNetworkConditions` before sending 8 MB. Without it the transfer finishes between two frames on loopback, the bar goes 0 → 100, and "progress moves" is untestable — the research doc flagged this before it was written.
- **Meeting page (`/meetings/[id]`)**: `page.tsx` is a server component that awaits `params` (Next 15 hands them over as a Promise) and passes the id to `meeting-view.tsx` (`'use client'`), which does the fetching. Four states, and the split matters: `loading`, `ready`, **`missing`** and `error`. `missing` is not an error — a meeting that isn't there is a normal answer to a stale or guessed link, and the API returns the same `404` for someone else's meeting as for one that never existed, so the page cannot tell them apart and must not try. A failed request is genuinely different and gets its own screen with a retry. Auth is gated in a mount effect exactly like the home view, and a `401` clears the dead token before redirecting. Meeting cards on the home list are whole-card links to this route.
- Path aliases in `tsconfig.json`: `@/*` maps to `src/*`; `@video-meetings/shared` maps directly to `packages/shared/src` (not a compiled `dist/`), so shared type changes are picked up without a separate build step.
- `next.config.ts` sets `transpilePackages: ['@video-meetings/shared']` so Next.js compiles the workspace package's TypeScript source directly rather than expecting pre-built JS.
- Consumes shared response types from `@video-meetings/shared` (e.g. `ApiResponse<T>`).
- `tsconfig.json` extends the root `tsconfig.base.json`, overriding module resolution for Next.js (`ESNext`/`bundler`, `jsx: "preserve"`, `noEmit: true` — Next's own toolchain handles emission).
- `.eslintrc.js` sets `root: true` (does not inherit the repo-root ESLint config) and extends `next/core-web-vitals` and `next/typescript`.
- **UI: HeroUI v3 + Playwright MCP** — see **`docs/architecture/frontend-ui.md`** for the full guide: HeroUI v3 setup/conventions (compound components, `onPress`, semantic variants, `next-themes` for light/dark — no `HeroUIProvider` in v3), the `validationBehavior="aria"` gotcha for any `Form`, and the Playwright MCP workflow for verifying layout/UI changes against a real browser. Wired up: `postcss.config.mjs`, `src/app/globals.css` (`@import 'tailwindcss'` then `@import '@heroui/styles'`), `src/app/providers.tsx` (`next-themes` `ThemeProvider`, no `HeroUIProvider`), `src/app/layout.tsx` imports `globals.css` and wraps `children` in `<Providers>`.
- **API client**: `src/lib/api/client.ts` exports `fetchJson<T>(path, options)` — the one shared `fetch` wrapper (plain `fetch`, no HTTP client library) all endpoint files build on; don't inline a new `fetch` call in a feature file, import this instead. It resolves the base URL from `NEXT_PUBLIC_API_URL` (see `.env.example`; defaults to `http://localhost:3001` in code if unset), guards `res.json()` against a non-JSON body (throws a plain `Error` with a clear message rather than an uncaught `SyntaxError`), and on a non-2xx response throws `ApiError` (`status`, `messages: string[]` — normalizing the API's two error shapes, a `string[]` from `ValidationPipe` 400s and a plain `string` from hand-thrown exceptions — plus an optional `field` when the API response includes one, e.g. the 409 duplicate-email conflict sets `field: "email"`; branch on `err.field`, not on `err.status` alone, to target a specific form field). On success it also verifies the response envelope actually has a `data` payload, throwing rather than silently returning `undefined` fields. Feature files (e.g. `src/lib/api/auth.ts`) additionally validate their own expected shape (e.g. `accessToken`/`user` are present) before returning. `src/lib/auth/token.ts` holds the `accessToken` `sessionStorage` helpers (`saveAccessToken`/`getAccessToken`, SSR-guarded) — the only place a token is persisted; reuse it rather than touching storage directly. `saveAccessToken` throws a distinct `StorageError` (not caught by the same handler as a failed API call) so a form can tell "registration succeeded but couldn't persist the session here" apart from "the request itself failed." `token.ts` also exposes `removeAccessToken` (logout) and `getUserEmailFromToken` (decodes the JWT payload client-side to greet the user — there is no `/me` endpoint; this is display-only, not a security check). `src/lib/api/meetings.ts` is the second endpoint file: `listMeetings(page)` sends the `Authorization: Bearer <token>` header (JWT-guarded route) and returns one page of meetings plus the authoritative `total`. It uses `fetchPaginated` (not `fetchJson`) — a sibling wrapper in `client.ts` that returns the full paginated envelope (`data`/`total`/`page`/`limit`) instead of just `data` — so the home list maps straight onto the API's own pagination and never holds more than one page in memory.

## UI change verification (mandatory)

Any change that affects rendered UI is not done until it has been visually verified:

- Visual verification must be done via the **Playwright MCP** tools — do not use any other browser/automation tool for this.
- Apply the **UI UX Pro Max** skill when designing or reviewing the change.
- The dev server is already running — do not start it yourself.
- A UI task is only considered complete after the visual check via Playwright MCP has passed.

## Keeping documentation current

When a change alters this app's architecture — new routing/layout structure, new shared-package usage, changed build config or path aliases — update this file (and the root `CLAUDE.md` if the change is monorepo-wide) in the same change.
